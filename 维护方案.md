### M-001：集成全局速率限制器到核心同步服务（P0）

**当前状态：**
- `rate_limiter.py` 已创建（CR-003 步骤1 ✅）
- `config.py` 参数已调整（CR-003 步骤2 ✅）
- `google_ads_service_account_sync.py` 第26行已 `import get_rate_limiter`，但**从未调用** ❌

**维护内容 — 4处 `time.sleep()` 逐一处理：**

| # | 位置 | 当前代码 | 处理方式 | 说明 |
|---|------|----------|----------|------|
| 1 | 第1082行 | `time.sleep(request_delay)` — 客户账号间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |
| 2 | 第1106行 | `time.sleep(delay)` — 配额限制退避（ResourceExhausted） | **保留不动** | 错误退避逻辑，不应走限速器 |
| 3 | 第1469行 | `time.sleep(batch_delay)` — MCC 批次间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 批次间限速，应由限速器控制 |
| 4 | 第1545行 | `time.sleep(settings.google_ads_request_delay_seconds)` — 历史数据每天间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |

**替换逻辑说明：**
- `acquire()` 返回 `True`：许可已获取，继续执行
- `acquire()` 返回 `False`：每日配额耗尽，跳出循环停止同步
- `acquire()` 内部会自动处理每分钟限速的阻塞等待

**蝴蝶效应预警：**
- 同步时间会从约 14 分钟延长到约 21 分钟（低峰期，不影响用户）
- 速率限制器阻塞等待时会释放锁，其他线程不受影响
- 第1106行保留原有退避逻辑，避免退避期间消耗限速器配额

### M-005：后端部署指令（部署运维）

**部署流程：**
代码修改完成并经 07 确认后，需要在服务器上执行以下操作：

1. SSH 连接服务器
2. 在**仓库根目录**拉取最新代码
3. 进入 backend 并**先激活虚拟环境**（未激活会报 `Exit 127`，找不到 uvicorn）
4. 停止当前后端服务
5. 启动新的后端服务（**单 worker，避免 SQLite 并发锁**）
6. 验证服务健康状态

**重要：** 必须先 `source venv/bin/activate` 再执行 `uvicorn`，虚拟环境路径：`~/Google-Data-Analysis/backend/venv`。

**部署指令：**
```bash
# 连接信息：主机 47.239.193.33，端口 22，用户 admin

# 1. 进入仓库根目录并拉取最新代码
cd ~/Google-Data-Analysis
git pull origin main

# 2. 进入后端目录并激活虚拟环境（必须，否则 Exit 127）
cd backend
source venv/bin/activate

# 3. 可选：更新依赖
pip install -q -r requirements.txt

# 4. 停止当前服务
pkill -f 'uvicorn.*app.main' || true
sleep 2

# 5. 启动后端服务（--workers 1 避免 SQLite 多进程写入冲突）
nohup uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1 > /tmp/backend.log 2>&1 &
sleep 3

# 6. 验证
curl -s http://127.0.0.1:8000/health
tail -n 15 /tmp/backend.log
```

**关于 --workers 1 的说明（VM-04）：**
- 当前后端使用 SQLite 数据库，SQLite 不支持多进程并发写入
- `--workers 2` 会导致两个进程同时写入数据库时出现 `database is locked` 错误
- 改为 `--workers 1` 后，所有请求由单进程处理，避免并发锁冲突
- 如果未来需要多 worker 性能，应先迁移到 PostgreSQL

**18-D：历史重复商家清理（一次性脚本）（MM-04 修正后）**

## 七、回滚方案（VL-03 + MM-03 补充）

如果维护上线后出现异常，按以下步骤回滚：

```bash
# 1. 停止当前服务
pkill -f "uvicorn app.main:app" || true
sleep 3

# 2. 回滚代码到上一个版本（建议使用 revert，不建议直接 reset --hard）
cd /home/admin/Google-Data-Analysis/backend
git log --oneline -5  # 查看最近5个提交
git revert HEAD        # 回滚最近一次提交

# 2.1 若本次包含数据库迁移（如 M-008 sender_id），执行对应降级迁移
# alembic downgrade -1

# 3. 重启服务
source venv/bin/activate
nohup python -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1 > /home/admin/backend.log 2>&1 &
sleep 5

# 4. 验证
curl -s http://localhost:8000/health
```
