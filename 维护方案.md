## 四、维护清单详细说明

### M-001：集成全局速率限制器到核心同步服务（P0）

**当前状态：**
- `rate_limiter.py` 已创建（CR-003 步骤1 ✅）
- `config.py` 参数已调整（CR-003 步骤2 ✅）
- `google_ads_service_account_sync.py` 第26行已 `import get_rate_limiter`，但**从未调用** ❌

**维护内容 — 4处 `time.sleep()` 逐一处理：**

| # | 位置 | 当前代码 | 处理方式 | 说明 |
|---|------|----------|----------|------|
| 1 | 第1082行 | `time.sleep(request_delay)` — 客户账号间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |
| 2 | 第1106行 | `time.sleep(delay)` — 配额限制退避（ResourceExhausted） | **保留不动** | 错误退避逻辑，不应走限速器 |
| 3 | 第1469行 | `time.sleep(batch_delay)` — MCC 批次间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 批次间限速，应由限速器控制 |
| 4 | 第1545行 | `time.sleep(settings.google_ads_request_delay_seconds)` — 历史数据每天间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |

**替换逻辑说明：**
- `acquire()` 返回 `True`：许可已获取，继续执行
- `acquire()` 返回 `False`：每日配额耗尽，跳出循环停止同步
- `acquire()` 内部会自动处理每分钟限速的阻塞等待

**蝴蝶效应预警：**
- 同步时间会从约 14 分钟延长到约 21 分钟（低峰期，不影响用户）
- 速率限制器阻塞等待时会释放锁，其他线程不受影响
- 第1106行保留原有退避逻辑，避免退避期间消耗限速器配额

### M-004：调整紧急补齐脚本延迟参数（P2）

**当前状态：**
- `emergency_backfill.py` 中 CUSTOMER_DELAY=2.0s, MCC_COOLDOWN=30s

**维护内容：**
- CUSTOMER_DELAY: 2.0 → 2.5 秒
- MCC_COOLDOWN: 30 → 60 秒

**计算依据（VL-02 补充）：**
- 每分钟限制 60 次，留 17% 余量 → 目标 50 次/分钟
- 60秒 / 50次 = 1.2 秒/次（理论最小间隔）
- CUSTOMER_DELAY = 2.5 秒 → 每分钟最多 24 次，留 60% 余量（紧急脚本需更保守）
- MCC_COOLDOWN = 60 秒 → 确保 MCC 切换时滑动窗口完全重置

**蝴蝶效应预警：**
- 紧急补齐脚本执行时间会延长约 25%
- 但大幅降低触发 429 的风险

### M-005：后端部署指令（部署运维）

**部署流程：**
代码修改完成并经 07 确认后，需要在服务器上执行以下操作：

1. SSH 连接服务器
2. 在**仓库根目录**拉取最新代码
3. 进入 backend 并**先激活虚拟环境**（未激活会报 `Exit 127`，找不到 uvicorn）
4. 停止当前后端服务
5. 启动新的后端服务（**单 worker，避免 SQLite 并发锁**）
6. 验证服务健康状态

**重要：** 必须先 `source venv/bin/activate` 再执行 `uvicorn`，虚拟环境路径：`~/Google-Data-Analysis/backend/venv`。

**部署指令：**
```bash
# 连接信息：主机 47.239.193.33，端口 22，用户 admin

# 1. 进入仓库根目录并拉取最新代码
cd ~/Google-Data-Analysis
git pull origin main

# 2. 进入后端目录并激活虚拟环境（必须，否则 Exit 127）
cd backend
source venv/bin/activate

# 3. 可选：更新依赖
pip install -q -r requirements.txt

# 4. 停止当前服务
pkill -f 'uvicorn.*app.main' || true
sleep 2

# 5. 启动后端服务（--workers 1 避免 SQLite 多进程写入冲突）
nohup uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1 > /tmp/backend.log 2>&1 &
sleep 3

# 6. 验证
curl -s http://127.0.0.1:8000/health
tail -n 15 /tmp/backend.log
```

**关于 --workers 1 的说明（VM-04）：**
- 当前后端使用 SQLite 数据库，SQLite 不支持多进程并发写入
- `--workers 2` 会导致两个进程同时写入数据库时出现 `database is locked` 错误
- 改为 `--workers 1` 后，所有请求由单进程处理，避免并发锁冲突
- 如果未来需要多 worker 性能，应先迁移到 PostgreSQL

### M-006：所有平台的广告数据匹配失败（P0 — Bug修复）

**问题现象：**
- wj06 在数据中心的"平台数据"标签页能看到 RW 的佣金数据（XCEL Solutions $16.54、Hornby UK $0.52）
- 但 L7D 分析中 RW 的广告系列无法匹配到联盟账号，导致 L7D 页面不显示 RW 的广告运营建议

**根因分析：**

数据库 `affiliate_platforms` 表中的实际数据：

| id | platform_code | platform_name |
|----|---------------|---------------|
| 1 | rw | Rewardoo |
| 2 | lb | LinkBux |
| 3 | lh | LinkHaiTao |
| 4 | cg | CollabGlow |
| 5 | pm | PartnerMatic |
| 6 | bsh | BrandSparkHub |
| 7 | cf | CreatorFlare |

代码中的匹配逻辑使用了错误的字段：

```python
# 当前代码（错误） — 用全名匹配缩写，永远不会成功
AffiliatePlatform.platform_name == platform_code
# "Rewardoo" == "RW" → False ❌
# "CollabGlow" == "CG" → False ❌
# "PartnerMatic" == "PM" → False ❌
```

**影响范围（VM-03 扩展）：**
- 此 Bug **影响所有平台**（RW/CG/PM/CF/LH/LB/BSH），不仅仅是 RW
- 所有用户的 L7D 分析中，广告数据与联盟账号的匹配全部失败
- 这意味着当前系统的 L7D 分析结果可能全部不准确

**修复方案（VS-02 + VM-02 修正后）：**

将匹配逻辑从 `platform_name` 改为 `platform_code`，对 Python 端变量做小写转换：

```python
# 修复后（正确）
AffiliatePlatform.platform_code == platform_code.lower()
# "rw" == "RW".lower() → "rw" == "rw" → True ✅
# "cg" == "CG".lower() → "cg" == "cg" → True ✅
```

**涉及修改位置（完整清单）：**

| # | 文件 | 行号 | 当前代码 | 修复为 |
|---|------|------|----------|--------|
| 1 | `api_analysis_service.py` | 第175行 | `AffiliatePlatform.platform_name == platform_code` | `AffiliatePlatform.platform_code == platform_code.lower()` |
| 2 | `api_only_analysis_service.py` | 第341行 | `account.platform.platform_name == platform_code.upper()` | `account.platform.platform_code == platform_code.lower()` |
| 3 | `api_only_analysis_service.py` | 第350行 | `AffiliatePlatform.platform_name == platform_code.upper()` | `AffiliatePlatform.platform_code == platform_code.lower()` |
| 4 | `api_only_analysis_service.py` | 第367行 | `AffiliatePlatform.platform_name == platform_code.upper()` | `AffiliatePlatform.platform_code == platform_code.lower()` |
