### M-001：集成全局速率限制器到核心同步服务（P0）

**当前状态：**
- `rate_limiter.py` 已创建（CR-003 步骤1 ✅）
- `config.py` 参数已调整（CR-003 步骤2 ✅）
- `google_ads_service_account_sync.py` 第26行已 `import get_rate_limiter`，但**从未调用** ❌

**维护内容 — 4处 `time.sleep()` 逐一处理：**

| # | 位置 | 当前代码 | 处理方式 | 说明 |
|---|------|----------|----------|------|
| 1 | 第1082行 | `time.sleep(request_delay)` — 客户账号间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |
| 2 | 第1106行 | `time.sleep(delay)` — 配额限制退避（ResourceExhausted） | **保留不动** | 错误退避逻辑，不应走限速器 |
| 3 | 第1469行 | `time.sleep(batch_delay)` — MCC 批次间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 批次间限速，应由限速器控制 |
| 4 | 第1545行 | `time.sleep(settings.google_ads_request_delay_seconds)` — 历史数据每天间延迟 | **替换为** `if not get_rate_limiter().acquire(): break` | 请求限速，应由限速器控制 |

**替换逻辑说明：**
- `acquire()` 返回 `True`：许可已获取，继续执行
- `acquire()` 返回 `False`：每日配额耗尽，跳出循环停止同步
- `acquire()` 内部会自动处理每分钟限速的阻塞等待

**蝴蝶效应预警：**
- 同步时间会从约 14 分钟延长到约 21 分钟（低峰期，不影响用户）
- 速率限制器阻塞等待时会释放锁，其他线程不受影响
- 第1106行保留原有退避逻辑，避免退避期间消耗限速器配额

### M-005：后端部署指令（部署运维）

**部署流程：**
代码修改完成并经 07 确认后，需要在服务器上执行以下操作：

1. SSH 连接服务器
2. 在**仓库根目录**拉取最新代码
3. 进入 backend 并**先激活虚拟环境**（未激活会报 `Exit 127`，找不到 uvicorn）
4. 停止当前后端服务
5. 启动新的后端服务（**单 worker，避免 SQLite 并发锁**）
6. 验证服务健康状态

**重要：** 必须先 `source venv/bin/activate` 再执行 `uvicorn`，虚拟环境路径：`~/Google-Data-Analysis/backend/venv`。

**部署指令：**
```bash
# 连接信息：主机 47.239.193.33，端口 22，用户 admin

# 1. 进入仓库根目录并拉取最新代码
cd ~/Google-Data-Analysis
git pull origin main

# 2. 进入后端目录并激活虚拟环境（必须，否则 Exit 127）
cd backend
source venv/bin/activate

# 3. 可选：更新依赖
pip install -q -r requirements.txt

# 4. 停止当前服务
pkill -f 'uvicorn.*app.main' || true
sleep 2

# 5. 启动后端服务（--workers 1 避免 SQLite 多进程写入冲突）
nohup uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1 > /tmp/backend.log 2>&1 &
sleep 3

# 6. 验证
curl -s http://127.0.0.1:8000/health
tail -n 15 /tmp/backend.log
```

**关于 --workers 1 的说明（VM-04）：**
- 当前后端使用 SQLite 数据库，SQLite 不支持多进程并发写入
- `--workers 2` 会导致两个进程同时写入数据库时出现 `database is locked` 错误
- 改为 `--workers 1` 后，所有请求由单进程处理，避免并发锁冲突
- 如果未来需要多 worker 性能，应先迁移到 PostgreSQL

### M-006：所有平台的广告数据匹配失败（P0 — Bug修复）

**问题现象：**
- wj06 在数据中心的"平台数据"标签页能看到 RW 的佣金数据（XCEL Solutions $16.54、Hornby UK $0.52）
- 但 L7D 分析中 RW 的广告系列无法匹配到联盟账号，导致 L7D 页面不显示 RW 的广告运营建议

**根因分析：**

数据库 `affiliate_platforms` 表中的实际数据：

| id | platform_code | platform_name |
|----|---------------|---------------|
| 1 | rw | Rewardoo |
| 2 | lb | LinkBux |
| 3 | lh | LinkHaiTao |
| 4 | cg | CollabGlow |
| 5 | pm | PartnerMatic |
| 6 | bsh | BrandSparkHub |
| 7 | cf | CreatorFlare |

代码中的匹配逻辑使用了错误的字段：

```python
# 当前代码（错误） — 用全名匹配缩写，永远不会成功
AffiliatePlatform.platform_name == platform_code
# "Rewardoo" == "RW" → False ❌
# "CollabGlow" == "CG" → False ❌
# "PartnerMatic" == "PM" → False ❌
```

**影响范围（VM-03 扩展）：**
- 此 Bug **影响所有平台**（RW/CG/PM/CF/LH/LB/BSH），不仅仅是 RW
- 所有用户的 L7D 分析中，广告数据与联盟账号的匹配全部失败
- 这意味着当前系统的 L7D 分析结果可能全部不准确

**修复方案（VS-02 + VM-02 修正后）：**

将匹配逻辑从 `platform_name` 改为 `platform_code`，对 Python 端变量做小写转换：

```python
# 修复后（正确）
AffiliatePlatform.platform_code == platform_code.lower()
# "rw" == "RW".lower() → "rw" == "rw" → True ✅
# "cg" == "CG".lower() → "cg" == "cg" → True ✅
```

### M-007：脚本模式 MCC 的 CPC 部署与出价管理恢复（P0 — 方案 A 混合模式）

**问题现象：**

- 用户无法执行：同步出价数据、转为人工 CPC、设置关键词 CPC、批量应用 CPC 建议等操作

**根因分析（现网行为 vs 目标行为）：**

| 维度 | 现网行为 | 目标行为（方案 A） |
|------|----------|-------------------|
| 数据同步 | `sync_mode=script` 走 Google Sheet | 保持不变 |
| 出价管理凭证选择 | `_create_client()` 受 `use_service_account` 分支控制 | 优先 service account，再回退 OAuth |
| 脚本模式可用性 | 若 `use_service_account=False` 且 OAuth 缺失则失败 | 脚本模式也可正常调用 Google Ads API |

说明：当前代码并非“天然 service account 优先”，而是先看 `use_service_account`。文档在 v1.2 已写成目标行为，v1.2b 纠正为“需要改造后达成”。

**服务器已具备条件：**
- `GOOGLE_ADS_SHARED_DEVELOPER_TOKEN` 已配置
- `GOOGLE_ADS_SERVICE_ACCOUNT_FILE` 或 `GOOGLE_ADS_SERVICE_ACCOUNT_JSON_BASE64` 已配置

**方案 A（混合模式）维护内容：**

1. **改造 `_create_client()` 凭证选择逻辑**（`google_ads_service_account_sync.py` 第 147-202 行）
   - 从“按 `use_service_account` 分支”改为“先尝试 service account（MCC 级别或全局）”
   - service account 不可用时，再回退 OAuth
   - 两者都不可用时返回明确错误：`MCC xxx 无可用的 API 凭证`

2. **出价管理 API 错误提示优化**（`bid_management.py`）
   - 当脚本模式 MCC 的 API 调用失败时，返回可执行提示：
     - 检查全局 service account 是否生效
     - 检查该 MCC 是否已授权给 service account
     - 检查开发者令牌对应 manager 账户权限

3. **前端 MCC 选择器标注**（`BidManagement.jsx`）
   - 在 MCC 下拉列表中，为 script 模式显示 `[脚本]`
   - 不禁用选择（混合模式目标即脚本模式可出价管理）

**架构示意：**

```
数据同步：Google Sheet (共享链接) → 后端 sync_mcc_from_sheet() → 数据库
出价管理：用户操作 → 后端 bid_management API → _create_client() → Google Ads API
```

**蝴蝶效应预警：**
- 改造后，所有 MCC（无论 sync_mode）都会优先尝试全局 service account
- 若某 MCC 与 service account 权限不匹配，API 会报权限错误，需在 Google Ads 侧补授权
- 若开发者令牌未覆盖该 manager 账户，调用仍会失败

**修复后验证步骤：**

2. 确认能成功拉取出价策略和关键词列表

### M-008：wj07 查看反馈并回复功能（P1 — 功能新增）

**问题现象：**
- wj07 作为维护人员，接收用户通过「提交反馈」投递的反馈
- 反馈以 `user_feedback` 类型通知形式存入 `notifications` 表，`user_id` 为 wj07
- wj07 只能在「消息通知」铃铛中看到反馈，与商家分配、拒付异常等通知混在一起
- 无专用反馈管理页面，无法按反馈类型筛选、无法查看完整内容
- **无回复能力**：wj07 无法对反馈进行回复，提交人无法收到维护人员的回应

**反馈隐私边界（MS-01 必须项）：**
- `user_feedback` 仅允许以下角色可见：
  1) 反馈提交人本人
  2) 维护人员 wj07
  3) manager（审阅与升级处理）
- 其他员工、组长默认不可见他人反馈
- `list_notifications` 在读取 `user_feedback` 时必须额外叠加隐私过滤，不得仅按 team 可见规则放开

**当前实现：**
- 后端 `POST /api/feedback`：提交反馈，创建 `Notification(user_id=wj07, type="user_feedback", ...)`
- 前端：用户菜单「提交反馈」弹窗，所有用户可见
- 通知内容含：提交人、角色、页面、类型、反馈内容（但未结构化存储 `submitter_id`）

**维护内容：**

1. **后端扩展（含隐私与回复链路）**
   - 新增 `GET /api/feedback`：wj07 获取反馈列表（仅 `type=user_feedback`），支持分页、按类型/时间排序
   - 新增 `POST /api/feedback/{notification_id}/reply`：wj07 回复某条反馈
   - 新增隐私过滤规则：普通通知走原规则，`user_feedback` 走专属可见范围规则
   - 回复时向原提交人发送通知：
     - 采用 `Notification.sender_id`（nullable）保存提交人，避免从文本反解析 user_id

2. **数据库迁移与历史兼容（MM-03）**
   - 新增迁移：`notifications.sender_id`（nullable, FK users.id）
   - 新反馈写入 `sender_id=current_user.id`
   - 历史反馈兼容策略：
     - 尝试按 `content` 中 username 一次性回填 `sender_id`
     - 回填失败的历史记录标记 `replyable=false`（接口层推导）
     - 前端展示“历史反馈缺少提交人信息，暂不可回复”

3. **前端新增**
   - 新增「反馈管理」页面（如 `/feedback-manage`），仅 wj07 可访问
   - 页面内容：反馈列表（类型、标题、提交人、时间、状态）、详情、回复输入框、提交回复按钮
   - 对 `replyable=false` 的历史项禁用回复按钮并显示提示

4. **入口与权限**
   - 在左侧导航或用户菜单中，仅 wj07 增加「反馈管理」入口
   - 未来可演进为 `maintainer` 角色（当前先按用户名白名单）

**蝴蝶效应预警：**
- 引入 `sender_id` 需要数据库迁移与回填脚本，发布窗口需包含迁移执行时间
- 历史反馈中部分记录可能无法回填提交人，需运营接受“只读不可回复”状态
