## 二、数据隔离架构专项审核（07 重点关注）

### 背景

07 指出：之前商家是经理分配（共用数据库），现在改为员工自助领取，涉及员工私人数据和共用数据的隔离。

### 现有架构分析

经代码验证，当前系统的数据分为两层：

**共用数据（全员可见）：**
- `affiliate_merchants` 表 — 商家目录（平台同步来的，所有人共享）
- `affiliate_transactions` 表 — 交易数据（平台同步来的）

**私有数据（绑定到特定用户）：**
- `google_mcc_accounts` 表 — MCC 账号（`user_id` 外键，每个员工自己的 MCC）
- `google_ads_api_data` 表 — 广告数据（`user_id` + `mcc_id` 外键，员工私有）
- `affiliate_accounts` 表 — 平台账号（`user_id` 外键，员工私有 Token）
- `merchant_assignments` 表 — 分配关系（`user_id` + `merchant_id`，桥接共用和私有）

### 方案对数据隔离的影响

员工自助领取后，广告创建流程需要同时访问：
1. **共用数据**：商家信息（`affiliate_merchants`）— 用于爬取商家网站、生成文案
2. **私有数据**：员工的 MCC/CID（`google_mcc_accounts` + `google_ads_api_data`）— 用于关键词研究和广告创建

**关键隔离要求：**
- 员工 A 的 MCC/CID 不能被员工 B 使用
- 员工 A 创建的广告只能出现在员工 A 的数据中心
- 空闲 CID 查找必须限定在当前员工自己的 MCC 下
- 广告创建后的 `google_campaign_id` 必须关联到正确的员工

### 方案评估

设计方案中 `find_available_cid()` 已正确使用 `GoogleMccAccount.user_id == user_id` 过滤，CID 隔离逻辑正确。但存在以下问题（见 S-16、M-45）。

## 三、发现问题

### 严重（S 级）— 1 项

#### S-16：权限模型断裂 — 领取 API 缺失权限降级设计

**现状：** 现有 `POST /api/merchant-assignments` 端点权限为 `get_current_manager_or_leader`（仅经理/组长可调用），`assigned_by` 字段记录分配操作者。

**方案设计：** 新增 `POST /api/merchant-assignments/claim` 端点供员工自助领取，但方案未说明：
1. 该端点的权限守卫是什么？（应为 `get_current_user`，即所有登录用户可调用）
2. `assigned_by` 字段如何处理？（员工自己领取时，`assigned_by` = `user_id`？还是设为 NULL？）
3. 现有的唯一约束 `(merchant_id, user_id, status)` 是否允许同一员工重复领取同一商家？
4. **经理/组长是否仍保留分配能力？** 如果保留，两套入口（经理分配 + 员工领取）如何共存？

**蝴蝶效应：** 如果不明确权限降级策略，可能导致：
- 员工调用现有分配 API 被 403 拒绝
- 或新领取 API 绕过了经理审批流程
- `assigned_by` 语义混乱，审计日志无法区分"经理分配"和"员工自领"

**建议：**
- 明确 claim API 权限为 `get_current_user`
- `assigned_by` 在自领时设为 `user_id`（自己分配给自己）
- 新增 `assignment_source` 字段（`"manager_assign"` / `"self_claim"`）区分来源
- 明确经理分配和员工领取是否并存，还是完全替代

### 中等（M 级）— 6 项

#### M-45：`find_available_cid()` 的 status 过滤值错误

**证据：** 代码验证发现 `google_ads_api_data.status` 字段存储的是**中文值**（`'已启用'`、`'已暂停'`、`'已移除'`），在 `google_ads_service_account_sync.py` L360-366 有明确的 `status_map` 转换：
```python
status_map = {'ENABLED': '已启用', 'PAUSED': '已暂停', 'REMOVED': '已移除', 'UNKNOWN': '未知'}
```

**方案代码（L257-259）：**
```python
GoogleAdsApiData.status.in_(['已启用', 'ENABLED'])
```

虽然方案同时包含了中文和英文值（防御性写法），但这暴露了设计人员对现有数据格式的不确定性。更重要的是：

**蝴蝶效应：** `all_cids` 查询没有过滤日期范围。如果一个 CID 在 3 个月前有已启用广告系列（现已暂停/移除），该 CID 仍会出现在 `busy_cids` 中（因为历史记录的 status 仍为 `'已启用'`），导致可用 CID 被错误排除。

**建议：** 
- 移除 `'ENABLED'`，只保留 `'已启用'`
- `busy_cids` 查询增加日期过滤：只看最近一次同步日期的数据（`GoogleAdsApiData.date == latest_sync_date`）

#### M-46：员工可能没有 MCC 账号 — `find_available_cid()` 缺少 None 检查

**现状代码（L245-248）：**
```python
mcc = db.query(GoogleMccAccount).filter(
    GoogleMccAccount.user_id == user_id,
    GoogleMccAccount.is_active == True
).first()
```

如果员工尚未绑定 MCC 账号，`mcc` 为 `None`，后续 `mcc.id` 会抛出 `AttributeError`。

**蝴蝶效应：** 新员工入职后立即尝试领取商家 → 500 错误 → 用户体验崩溃。

**建议：** 
- `mcc` 为 None 时返回明确错误："请先在「MCC账号」页面绑定您的 Google Ads MCC 账号"
- 前端领取按钮应先检查员工是否有 MCC，无则禁用并提示

#### M-47：员工可能有多个 MCC — `.first()` 只取第一个

**现状：** `GoogleMccAccount` 表的 `user_id` 没有唯一约束，一个员工可以绑定多个 MCC。`.first()` 只返回第一个，可能不是员工想用的那个。

**蝴蝶效应：** 员工有 2 个 MCC（MCC-A 和 MCC-B），系统默认用 MCC-A 的空闲 CID，但员工想用 MCC-B 的 → 广告创建在错误的 MCC 下。

**建议：** 
- Step 5 已有"选择 MCC/CID"，但 Step 1 的关键词研究也需要 CID
- 应在 Step 1 之前让员工选择 MCC，或在领取弹窗中选择
- `find_available_cid()` 应接受 `mcc_id` 参数而非 `user_id`

#### M-48：广告创建后 `google_ads_api_data` 数据同步缺口

**现状：** `google_ads_api_data` 表的数据来源是每日定时同步（`sync_all_mccs`），不是实时的。

**蝴蝶效应：** 员工在 10:00 创建了广告 → `MerchantAssignment.google_campaign_id` 已记录 → 但 `google_ads_api_data` 要到次日凌晨同步才有这条广告的数据 → 员工当天在数据中心看不到刚创建的广告 → 以为创建失败。

**建议：** 
- 广告创建成功后，立即向 `google_ads_api_data` 插入一条初始记录（status='已启用'，费用/点击等为 0）
- 或在数据中心增加"刚创建的广告"提示，说明数据将在次日同步后显示

#### M-49：`create_full_campaign()` 缺少错误处理和回滚设计

**方案代码（L321-338）：** `GoogleAdsCreator.create_full_campaign()` 只描述了成功路径，未说明：
1. Mutate 部分失败时的处理（如预算创建成功但广告组创建失败）
2. Google Ads API 返回 `QUOTA_EXCEEDED` 时的处理
3. 创建成功后 `MerchantAssignment.google_campaign_id` 的更新时机
4. 网络超时时是否需要查询确认（避免重复创建）

**蝴蝶效应：** 打包 Mutate 本身是原子的（全成功或全失败），但如果 API 调用超时，客户端不知道是否成功 → 重试可能创建重复广告系列。

**建议：** 
- 增加幂等性设计：创建前先查询是否已存在同名广告系列
- 增加 `QUOTA_EXCEEDED` 的友好提示
- 明确 `MerchantAssignment` 更新和 Mutate 调用的事务边界

#### M-50：测试商家看板 API 数据来源不明确

**方案（Section 5.1）：** 4 个 API 端点（overview/list/analysis/ai-review），但未说明数据如何聚合：
- 广告数据来自 `google_ads_api_data`（按 `google_campaign_id` 关联）
- 平台佣金数据来自 `affiliate_transactions`（按 `merchant_id` 关联）
- 但 `affiliate_transactions` 是按平台账号同步的，不区分测试/普通模式

**蝴蝶效应：** 同一商家如果同时被员工 A（测试模式）和员工 B（普通模式）领取，平台佣金无法区分归属 → 测试看板的佣金数据不准确。

**建议：** 
- 明确测试看板的佣金数据是否只看广告侧（`google_ads_api_data` 的 cost/clicks），还是也包含平台佣金
- 如果包含平台佣金，需要说明如何按 `assignment` 拆分

### 轻微（L 级）— 4 项

#### L-36：`generate_keyword_ideas()` 参数 `db` 位置不规范

**方案代码（L266-268）：**
```python
def generate_keyword_ideas(self, user_id, merchant_url, seed_keywords, country_code, language_code, db)
```

`db: Session` 作为最后一个参数不符合项目惯例。现有代码中 `db` 通常在 `__init__` 中注入（如 `GoogleAdsServiceAccountSync.__init__(self, db)`）。

**建议：** `KeywordPlanService.__init__(self, db)` 注入 db，方法签名中移除 db 参数。

#### L-37：`_create_client()` 调用方式与现有代码不一致

**方案代码（L274-275）：**
```python
sync_service = GoogleAdsServiceAccountSync()
client = sync_service._create_client(mcc)
```

**现有代码：** `GoogleAdsServiceAccountSync.__init__(self, db: Session)` 需要 `db` 参数。方案中 `GoogleAdsServiceAccountSync()` 无参构造会报错。

**建议：** 改为 `sync_service = GoogleAdsServiceAccountSync(db)` 或直接在 `KeywordPlanService` 中复用 `_create_client` 逻辑。

#### L-38：方案缺少 Alembic 迁移说明

**方案（Section 2.1）：** `MerchantAssignment` 新增 5 个字段（`mode`、`google_campaign_id`、`google_customer_id`、`daily_budget`、`target_country`），但未提供 Alembic 迁移脚本或迁移步骤。

**建议：** 补充迁移命令和字段默认值说明（特别是 `mode` 默认值 `"normal"` 对现有数据的影响）。

#### L-39：文件变更清单缺失

方案未提供完整的文件变更清单（新建/修改文件列表），不符合设计方案的一贯格式。

**建议：** 补充文件变更清单，列出所有新建和修改的文件。

## 四、可行性与可靠性评估

### 4.1 技术可行性

| 模块 | 可行性 | 评估 |
|------|--------|------|
| KeywordPlanIdeaService | ✅ 高 | Google Ads API 官方支持，现有 `_create_client()` 认证基础设施可复用 |
| 打包 Mutate 创建广告 | ✅ 高 | Google Ads API 官方推荐的批量操作方式，临时 ID 机制成熟 |
| AI 广告素材生成 | ✅ 高 | 复用现有 `_call_with_fallback()` 基础设施，Prompt 逻辑合理 |
| 空闲 CID 本地查找 | ⚠️ 中 | 逻辑正确但需修复日期过滤问题（M-45） |
| 员工自助领取 | ⚠️ 中 | 权限模型需重新设计（S-16） |
| 测试看板数据聚合 | ⚠️ 中 | 佣金归属逻辑需明确（M-50） |

### 4.2 可靠性评估

| 风险点 | 级别 | 说明 |
|--------|------|------|
| API 额度消耗 | 低 | 2 次/商家，日额度 2880 次，可处理 1440 个商家，充裕 |
| 广告创建失败 | 中 | 打包 Mutate 是原子操作，但需处理超时和重复创建（M-49） |
| 数据隔离 | 中 | CID 查找已按 user_id 隔离，但多 MCC 场景需处理（M-47） |
| 数据同步延迟 | 中 | 新创建广告当天不可见（M-48） |
| 权限混乱 | 高 | 经理分配和员工领取并存时的权限边界不清（S-16） |

### 4.3 数据隔离架构总结

| 数据层 | 类型 | 隔离方式 | 方案处理 |
|--------|------|---------|---------|
| 商家目录 | 共用 | 全员可见 | ✅ 正确，搜索页展示所有商家 |
| MCC/CID | 私有 | `user_id` 外键 | ⚠️ 基本正确，但多 MCC 场景需处理 |
| 广告数据 | 私有 | `user_id` + `mcc_id` 外键 | ⚠️ 需处理创建后同步延迟 |
| 平台账号/Token | 私有 | `user_id` 外键 | ✅ 不涉及本方案 |
| 分配关系 | 桥接 | `user_id` + `merchant_id` | ⚠️ 需明确领取来源标记 |
| 平台佣金 | 共用 | 按平台账号同步 | ⚠️ 测试/普通模式佣金无法区分 |

## 六、给 07 的审核清单

07，以下是第二十七轮审核总结：

**方案整体评价：** 技术选型优秀（Google Ads API 替代 SEMrush 零成本、打包 Mutate 省额度、AI 素材生成复用现有基础设施），但在权限模型和数据隔离方面存在架构级缺陷。

**你特别关注的数据隔离问题：**
- 商家目录（共用）→ 员工 MCC/CID（私有）→ 分配关系（桥接）的三层架构是正确的
- `find_available_cid()` 已按 `user_id` 隔离，员工 A 不会用到员工 B 的 CID
- 但存在 3 个隔离漏洞：多 MCC 选择（M-47）、创建后数据同步延迟（M-48）、测试/普通佣金混淆（M-50）

**最关键的问题是 S-16：** 从"经理分配"到"员工领取"是权限模型的根本性变更，方案没有说清楚两套机制如何共存、`assigned_by` 语义如何变化、是否需要新增来源标记字段。这个不解决，后续所有功能都建立在不稳定的地基上。

**建议设计人员优先处理 S-16 和 M-45~M-47，修正后提交复审。**
