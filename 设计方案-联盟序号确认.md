# 联盟序号（account_index）用户确认机制

> 立项人：07 ｜ 执行：AI ｜ 立项日期：2026-07-15
> 说明：根目录 `设计方案.md` 中文已被编码事故损坏（约 19 万个 `?`），本功能单独立档，待主设计方案修复后并入。

---

## 1. 背景与病因

- 广告系列名里的平台段序号（如 `LH1` / `PM2`）通过 `platform_connections.account_index`（D-168，07 规则：同用户同平台第 1 个账号=1、第 2 个=2，删号补最小空缺位次）映射到具体联盟连接。
- 现状：`account_index` 在**新建连接时自动分配**，全库 159 个在用连接均有值、无 NULL、无重复。
- 病因：**自动分配的序号未必等于用户心里的编号**。账号删除重加、创建顺序等会让 `account_index` 与用户意图错位，导致广告名写 `PM1` 却被路由到系统认定的另一个连接。

### 真实事故（2026-07-15 定位）

- yz04 的 LH 账号 `lanshao` 于 06-26 删旧连接 267、重加新连接 268；但商家 `connection_campaign_links` 链接键停留在旧连接 267，广告/订单已归到 268。
- `pickCampaignAffiliateLink(268)` 找不到 268 的链接、按防串号规则返回空串 → 换链补货与自动刷点击**双双静默跳过**。
- 表现：mohawk/mgmresorts/ticombo/johnvarvatos 联盟点击 07-12 起齐刷刷坍塌到 1/天。
- 全站同类错配共 17 条广告（涉 wj04/wj05/wj08/wj10/wj111/yz02/yz04）。

---

## 2. 目标

把"联盟序号"从系统自动分配升级为**用户强制确认**，并在确认后自动修复错配的广告归属与链接键。

---

## 3. 需求定版（07 拍板）

| 决策点 | 结论 |
|---|---|
| 弹窗位置 | **登录后全局强制弹窗**：只要有未确认的多连接平台，进系统必须先确认才能操作 |
| 单连接平台 | 自动 `account_index=1` 且**直接标记为已确认**，永不打扰 |
| 确认后 | **自动重排**受影响广告归属（广告名序号段 → 同序号连接）+ 迁移 `connection_campaign_links` 链接键 |
| 序号默认值 | 用当前 `account_index` **预填**，用户核对/微调即可 |

---

## 4. 判定"未确认"的机制

现有 `account_index` 恒有值，无法用 NULL 判定。新增列：

- `platform_connections.account_index_confirmed_at DATETIME NULL`
  - NULL = 未确认（含存量自动分配）
  - 有值 = 用户已确认（或单连接平台被系统自动确认）

**需要弹窗** = 用户存在某平台，该平台在用连接数 ≥ 2，且其中存在 `account_index_confirmed_at IS NULL` 的连接。

### 5.1 状态 API `GET /api/user/account-index/status`

- 副作用：把"单连接平台"的连接自动补 `account_index=1` 且 `confirmed_at=now()`（幂等）。
- 返回：`needsConfirm: boolean` + `platforms: [{ platform, connections: [{ id, account_name, account_index, confirmed }] }]`（仅多连接且未确认的平台）。

### 5.2 保存 API `POST /api/user/account-index/confirm`

- 入参：`{ platform, assignments: [{ connectionId, accountIndex }] }`（可多平台批量）。
- 校验：同平台内序号唯一、连续 1..N、覆盖该平台全部在用连接、无空缺。
- 写入：`account_index` + `account_index_confirmed_at=now()`。
- 成功后调用自动重排模块（同平台）。

### 5.3 自动重排模块 `account-index-reassign.ts`

- 对指定 (user, platform)：
  1. 按确认后的 `account_index` 建 `序号 → connId` 映射。
  2. 遍历该用户该平台的在投广告，按广告名平台段序号（`parseCampaignNameFull`）**强制**重排 `campaigns.platform_connection_id`（覆盖，非仅补空）。
  3. 对每条受影响商家，把 `connection_campaign_links` 里"旧 connId"的链接键迁到"新 connId"（复用 platforms DELETE 的 twin-key 迁移思路；仅同账号安全迁移）。
  4. 全程 before/after 对比日志，dry-run 模式只输出不写。

### 5.4 前端全局弹窗

- `UserLayout` 挂载后拉 `/api/user/account-index/status`；`needsConfirm` 为真则弹**不可关闭**模态框。
- 按平台分组，每连接一个序号下拉（预填当前值），保存校验通过后提交确认 API，成功后关闭并 `mutate` 状态。

### 5.5 生产库只读 dry-run 结果（2026-07-15，模拟「全员按当前序号确认」）

在服务器以只读脚本模拟全量重排（排除 REMOVED 广告），结果：

| 链接键自动迁移 | **161 条** | 含 **yz04 LH 267→268 全量 45 个商家（mohawk/johnvarvatos/ticombo 等本次事故全部命中）**、wj08 PM 14→173、wj08 LH 33→162 |

安全设计定版（写路径规则）：
- 目标连接已有链接（per-conn 键 或 主连接+主链接）→ 不动。
- 来源键属**已删除**连接 → 搬走（清残留）；属**在用**连接 → 复制保留（他号广告可能仍在用）。
- 主链接挂在同账号旧主连接 → 只把链接**复制**进目标 per-conn 键，不改商家 `platform_connection_id` 指针。
- 账号名比较忽略大小写与首尾空白；跨账号一律不迁。

## 6. 风险与约束

- **写风险集中在自动重排**（覆盖 `platform_connection_id` + 改 `connection_campaign_links`）。措施：仅同平台内、按序号严格映射、同账号才迁链接键、全量日志、上线前影子库 dry-run。
- Migration **只在本地写文件**，提交后由 CI 在服务器 `prisma migrate deploy`，**绝不本地对生产库执行**。
- 迁移不做数据回填 `confirmed_at`（存量一律视为未确认），让多连接用户上线后首次确认一遍。

## 9. 附：落地洗参广告主 no_tracking 误判修复（D-182，2026-07-16）

现象：yz04 `168-RW2-gilt-US-0715-136861` 明明有有效链接，换链接却「跟链失败」。

根因（服务器逐跳跟链定位）：gilt 链路为 rewardoo/track → flexlinkspro → **prf.hn(Partnerize)** →
`gilt.com/?clickref=...&utm_source=...`（此跳追踪参数已生成、点击已注册）→ gilt 前端把 URL 洗成干净的
`gilt.com/boutique/`（query 清空）。CRM 取参口径是「取**最终落地页** query 作 suffix」，最终页无 query
→ 判 `no_tracking`（跟链失败）。但点击其实在「广告主域名带参那一跳」已注册成功，属**误判**。
同类受影响：日志中同为 no_tracking 的 lusetabeauty、bradfordexchangechecks 等走 Partnerize 类联盟的商家。

修复（`affiliate-link-resolver.ts`）：
- 新增 `salvageTrackingFromChain`：最终落地页无 query 时，在跳转链里**回溯**与最终落地页**同根域名**、
  且带**已知联盟追踪参数**（clickref/clid/irclickid/ranMID/utm_*/subid/cjevent 等，见 `TRACKING_QUERY_KEYS`）
  的那一跳，取其 query 串作 suffix。
- 安全约束：① 必须同根域名（该参数是广告主自己域名接受过的，附到 Google finalUrl 安全）；
  ② 必须命中已知联盟追踪键（避免误取 ?locale=en 之类站内参数）；③ 仅在原判 no_tracking 时兜底，
  不改变 ok / forbidden_network / resolve_failed / 跳板中转域名 等既有判定路径。
- HTTP 与浏览器兜底两条路径都经过 `evaluate`，故两条路径同时生效。

## 10. RW 转化率反推自查（D-184，2026-07-16，第一步·只读）

背景（服务器实测确认）：07 提供了 Rewardoo `click_details` 接口文档，实测用 yz04 两个 RW token
（conn 279/273）在多个时段调用（含 7/15 16:00 那个真实成交小时），**恒返回 `{status:200, payload:{total:0,list:[]}}`**。
根因：**RW 是聚合型联盟**（gilt 走 Partnerize、olaplex 走 Rakuten…），点击在**下游联盟域名**注册，
RW 只做中转、只回传订单，故「订单可见、点击天生不可见」。`click_details` 对我方 medium token 无点击数据，
非调用姿势问题——与 `platform-api.ts` 既有注释一致。

静默失败全站扫描结论（近3天，yz04）：**可测平台（CG/LH/LB/MUI/PM/BSH）零真静默失败**，刷出去的点击
联盟到账数均 ≥ 刷成功数；那 302 次「刷了联盟 0」全部落在 RW（测量盲区，非白刷——RW 订单正常回传证明链路通）。
记录在案失败：no_tracking 144（CG1 78 + BSH1 66，D-182 对症）、生成超时 61、跳板未跟到 2。

反推口径（不依赖 RW 点击接口）：
> **联盟侧可见点击 ≈ Google 真实点击（ads_daily_stats）+ 我方刷点击（kyads_click_task_items success）**
> （两者都打在同一条 RW 跟链上）
> **反推转化率 = RW订单 ÷ (Google点击 + 刷点击)**，目标 ≤ 用户设定 max_pct（默认 10%）。

近3天实测抓出问题：lenstore(279) 76 Google点击 + 0 刷 / 13 单 = **17.1% 超标**（漏刷）；
根因是现有 RW 补刷仅「当天有新订单」才触发、且真实点击当 0 算——lenstore 13 单集中 7/15、7/16 无新单即被跳过。
samsclub(279) 因当天持续出单被刷到 2.5%。**同为 RW，一个刷爆一个漏刷。**（第二步再议是否把 Google 点击喂进补刷引擎。）

实现（第一步·纯只读，不触发任何刷点击/写库）：
- 新增 `GET /api/user/link-exchange/rw-reconcile?days=3|7|14`：按登录用户的 RW（active+paused、有
  google_campaign_id）广告，按 (连接, 商家) 聚合 Google点击 / 刷成功 / RW订单，算反推转化率，超标标红。
  时区：订单/刷点击按 CST 自然日（DATETIME UTC 窗口），ads_daily_stats.date 按 CST 归档（DATE 列 UTC 比较）。
- 换链接管理页新增「RW核对」Tab（`src/app/user/link-exchange/page.tsx`）：表格展示 + 超标行标红 +
  Tab 角标显示超标数 + 顶部说明反推口径与 RW 黑盒背景；「有单零点击」单列最坏标记。
- 不改任何写路径 / 补刷逻辑，零生产风险。
