## 二、OPT-010 L7D AI 深度分析增强

### 2.1 功能描述

将 L7D 分析页面的 AI 报告能力从 Gemini 通用文本报告升级为 Claude 结构化多维深度分析。参考业界 Claude Skills（google-ads-report 分析框架），新增三大原生分析模块：异常检测、CPA 诊断、预算效率建模，直接替换现有 AI 报告列。

**核心问题：**

1. 现有 AI 报告使用 Gemini + 通用提示词，分析深度不足
2. 仅传入当前 7 天汇总表，无前后对比数据（无法做趋势分析）
3. 无异常检测能力（不知道什么是"正常"）
4. 操作指令为固定阈值规则，缺乏多因素归因

### 2.2 AI 引擎切换

#### 2.2.1 当前架构

```
L7D数据 → GeminiService.generate_operation_report() → 纯文本报告 → ai_report字段
```

#### 2.2.2 目标架构

```
L7D数据 + 上期数据 + 异常检测 + CPA诊断 + 预算分析
    ↓
ClaudeAnalysisService.generate_l7d_report()
    ↓（内部组合调用 ClaudeService._call_api_with_fallback()）
    ↓
结构化分析报告（JSON）→ 前端结构化卡片渲染
    ↓（整体失败时）
GeminiService（降级备选）
```

#### 2.2.3 ClaudeAnalysisService 与现有 ClaudeService 的关系（S-13 闭环）

**采用组合模式（Composition），不继承、不独立实现 API 调用。**

现网 `backend/app/services/claude_service.py` 已有完整的 `ClaudeService` 类，包含：
- Anthropic 客户端初始化（含哈基米 `sk-` Key 自动适配）
- `_call_api_with_fallback()` 主/备模型切换
- 错误处理与日志

`ClaudeAnalysisService` 的职责定位：

| 类 | 职责 | 文件 |
|-----|------|------|
| `ClaudeService` | 通用 Claude API 封装（客户端、fallback、Tool Use） | `claude_service.py`（已存在，不改） |
| `ClaudeAnalysisService` | L7D 专用分析逻辑（提示词构建、数据组装、结果解析） | `claude_analysis_service.py`（新建） |

组合方式：

```python
class ClaudeAnalysisService:
    def __init__(self, claude_service: ClaudeService):
        self.claude = claude_service  # 组合，复用同一个 Anthropic 客户端

    def generate_l7d_report(self, data, anomalies, cpa_diagnosis, budget_analysis):
        prompt = self._build_prompt(data, anomalies, cpa_diagnosis, budget_analysis)
        response = self.claude._call_api_with_fallback(
            max_tokens=4096,
            messages=[{"role": "user", "content": prompt}]
        )
        return self._parse_response(response)
```

初始化方式（`api_analysis_service.py` 中）：

```python
claude_svc = ClaudeService(api_key=settings.CLAUDE_API_KEY, ...)  # 复用已有实例
analysis_svc = ClaudeAnalysisService(claude_service=claude_svc)   # 组合注入
```

**共存策略：**
- 整个系统只有一个 `ClaudeService` 实例（由 `config.py` 配置初始化）
- `ClaudeAnalysisService` 不创建新的 Anthropic 客户端，不读取 `.env`
- `.env` 中同一个 `CLAUDE_API_KEY` 只被 `ClaudeService` 读取一次

#### 2.2.4 Claude API 配置

复用现有 `config.py` 中已定义的 Claude 配置：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `CLAUDE_API_KEY` | 从 `.env` 读取 | 哈基米代理 API Key |
| `CLAUDE_BASE_URL` | `https://api.gemai.cc` | 哈基米代理地址 |
| `CLAUDE_MODEL` | `[特价B]claude-sonnet-4-20250514` | 主模型 |
| `CLAUDE_MODEL_FALLBACK` | `[特价B]claude-opus-4-5-20251101` | 备用模型 |

Key 存储方式与 Gemini 一致：`.env` 文件 → `Settings` 类读取 → 运行时内存使用，不写入日志或数据库。

### 2.3 数据增强：前后双周期对比

#### 2.3.1 改造位置

`api_analysis_service.py` 的 `generate_l7d_analysis()` 方法。

#### 2.3.2 改造逻辑

当前仅查询 `[end_date - 6, end_date]` 的 7 天数据。新增：

| 查询窗口 | 日期范围 | 用途 |
|----------|----------|------|
| 当前期（current） | `end_date - 6` ~ `end_date` | 现有逻辑不变 |
| 上期（previous） | `end_date - 13` ~ `end_date - 7` | 用于周环比对比 |
| 历史（history） | `end_date - 29` ~ `end_date` | 用于趋势分析和预算建模 |

#### 2.3.3 周环比指标

为每个 campaign 计算以下环比变化率：

| 指标 | 计算方式 |
|------|----------|
| cost_change_pct | (当前花费 - 上期花费) / 上期花费 |
| clicks_change_pct | (当前点击 - 上期点击) / 上期点击 |
| cpc_change_pct | (当前CPC - 上期CPC) / 上期CPC |
| commission_change_pct | (当前佣金 - 上期佣金) / 上期佣金 |
| roi_change | 当前ROI - 上期ROI（绝对值变化） |

上期数据为空时，标记为"首周数据（无对比基线）"。

#### 2.3.4 逐日粒度数据查询（M-24 + L-17 + M-28 闭环）

异常检测（2.4）需要 14 天逐日指标，预算效率分析（2.6）需要 30 天逐日花费数据。统一查询 30 天，异常检测取最近 14 天子集。

### 2.4 异常检测模块

#### 2.4.1 算法

基于 14 天滚动均值 + 标准差的统计异常检测。

| 检测维度 | 数据来源 | 异常判定 |
|----------|----------|----------|
| CPC | `GoogleAdsApiData.cpc` | 偏离均值超 2 倍标准差 |
| CTR | `clicks / impressions` | 偏离均值超 2 倍标准差 |
| 日花费 | `GoogleAdsApiData.cost` | 偏离均值超 2 倍标准差 |
| IS Budget 丢失 | `GoogleAdsApiData.is_budget_lost` | 偏离均值超 2 倍标准差 |
| IS Rank 丢失 | `GoogleAdsApiData.is_rank_lost` | 偏离均值超 2 倍标准差 |

#### 2.4.2 严重程度分级

| 级别 | 判定条件 | 颜色 |
|------|----------|------|
| critical | 偏离 > 3 倍标准差 | 红色 |
| warning | 偏离 > 2 倍标准差 | 黄色 |
| info | 偏离 > 1.5 倍标准差 | 蓝色 |

#### 2.4.3 原因推断规则

| 异常组合 | 推断原因 |
|----------|----------|
| CPC 飙升 + IS Rank 丢失上升 | 竞争对手加价 |
| CTR 下降 + 展示量不变 | 广告疲劳 / 相关性下降 |
| 花费激增 + 点击不变 | CPC 异常升高 |
| IS Budget 丢失暴涨 | 预算不足 |
| 花费骤降 + 展示量骤降 | 广告被暂停或审核拒绝 |

### 2.5 CPA 诊断模块

#### 2.5.1 适用条件

#### 2.5.2 orders 数据来源（M-25 + M-29 闭环）

**上期 orders/commission：新增 `merchant_previous_data` 查询。**（M-29 闭环）

CPA 诊断需要 `old_cpa = previous_cost / previous_orders`，因此需要上期（`end_date-13` ~ `end_date-7`）的 orders 和 commission。复用 `merchant_l7d_data` 的查询逻辑，日期范围改为上期窗口：

```python
merchant_previous_data = {}
for uid in user_ids:
    prev_txns = db.query(
        AffiliateTransaction.merchant_id,
        func.sum(AffiliateTransaction.commission_amount).label('total_commission'),
        func.count(AffiliateTransaction.id).label('total_orders')
    ).filter(
        AffiliateTransaction.user_id == uid,
        AffiliateTransaction.merchant_id.isnot(None),
        AffiliateTransaction.merchant_id != 'None',
        func.date(AffiliateTransaction.transaction_time) >= end_date - timedelta(days=13),
        func.date(AffiliateTransaction.transaction_time) <= end_date - timedelta(days=7)
    ).group_by(AffiliateTransaction.merchant_id).all()

    for txn in prev_txns:
        mid = str(txn.merchant_id)
        merchant_previous_data[(uid, mid)] = {
            "commission": float(txn.total_commission or 0),
            "orders": int(txn.total_orders or 0)
        }
```

CPA 诊断模块使用方式：

```python
current = merchant_l7d_data.get((user_id, mid), {})
previous = merchant_previous_data.get((user_id, mid), {})
current_orders = current.get("orders", 0)
previous_orders = previous.get("orders", 0)
```

**防除零保护：** 当期或上期 `orders == 0` 时，该 campaign 不参与 CPA 归因计算，在结果中标记 `"excluded_reason": "当期或上期无订单数据"`。

#### 2.5.3 归因算法

对比当前 7 天 vs 上一个 7 天，拆解 CPA 变化为以下因素：

| 归因因素 | 计算方式 | 说明 |
|----------|----------|------|
| CPC 变化 | `(new_cpc - old_cpc) / old_cpa` | CPC 上升直接推高单次获客成本 |
| CVR 变化 | `old_cpc / new_cvr - old_cpc / old_cvr` | 转化率下降导致需要更多点击 |
| 预算分配 | `Σ(share_change × campaign_cpa)` | 低效 campaign 占比上升 |

其中：
- CPA = cost / orders（orders 来源见 2.5.2，orders > 0 时有效）
- CVR = orders / clicks（clicks > 0）

### 2.6 预算效率分析

#### 2.6.1 算法

基于过去 30 天各 campaign 的日花费 vs 日佣金数据，拟合花费-佣金关系曲线。

| 计算项 | 方法 |
|--------|------|
| 边际效率 | `delta_commission / delta_cost`（花费每增加 $1 带来的佣金增量） |
| 效率拐点 | 边际效率 < 1 的花费水平（继续加预算会亏损） |
| 推荐预算 | 边际效率 >= 保守阈值（0.864）时的最大花费（L-15 闭环：0.72 为平台平均佣金率基线，即每 $1 花费平均带来 $0.72 佣金；×1.2 为 20% 安全系数，确保推荐预算仍处于盈利区间） |

#### 2.6.2 输出结构

```json
{
  "budget_analysis": [
    {
      "campaign_name": "BSH US [wj02]",
      "current_daily_budget": 50.00,
      "recommended_daily_budget": 65.00,
      "change": "+30%",
      "reason": "边际效率1.8，IS Budget丢失35%，有增长空间",
      "projected_roi_change": "+12%"
    }
  ]
}
```

#### 2.9.2 降级策略

Claude API 调用失败时：
1. 记录错误日志
2. 自动降级到 Gemini（使用现有 `generate_operation_report()` 逻辑）
3. 返回结果中标记 `ai_engine: "gemini_fallback"`，前端按旧格式渲染

### 2.10 安全约束

| 约束 | 说明 |
|------|------|
| API Key 存储 | `.env` 文件，不入库，不写日志 |
| 数据传输 | 仅传广告指标数据给 Claude，不传用户个人信息 |
| 频率控制 | 每用户每小时最多触发 5 次手动 L7D 分析。实现方式（L-16 闭环）：后端 in-memory `defaultdict(list)` 存储每用户的请求时间戳列表，每次请求时清理超过 1 小时的旧记录并检查长度；服务重启后计数器自动重置（可接受，因非安全关键限制） |
| 降级保障 | Claude 不可用时自动降级到 Gemini，不影响核心功能 |
