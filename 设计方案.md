## 五、设计原则与规范

1. **严谨性**：所有功能改动必须先写方案、经 07 确认，再编码实施。
2. **可追溯**：每次变更在「设计讨论表」补充记录，保留完整沟通历史。
3. **零报错**：代码提交前需通过 ESLint、TypeScript 类型检查、Prisma 校验，确保无报错。
4. **Prisma 规范**：数据结构变更必须走 `prisma migrate` 流程，禁止直接操作数据库。
5. **最小改动**：在满足需求前提下，尽可能少改动现有逻辑，避免引入未知风险。
6. **UI 一致性**：前端统一使用 Ant Design 组件库，保持整体风格一致。

### 9.1 需求背景（07 验收要求）

| 编号 | 需求 | 说明 |
|------|------|------|
| REQ-1 | 前端固定展示 **20 张**图片 | 后端可爬取 30/40/50 张，只要最终精选出 20 张即可 |

---

### 9.2 现状问题分析

**当前流程（有问题的）：**

```
后端爬取 → collectImages（目标60张原始）
         ↓
         selectBestImages（URL黑名单过滤 + HEAD检查 + 评分排序）
         ↓
         返回最多 40 张给前端（前端显示 20 张）
```

**问题清单：**

| 编号 | 问题 | 影响 |
|------|------|------|
| P-1 | **无 OCR 过滤**：仅靠 URL 路径关键词过滤含文字图片，误判率高 | 大量促销 banner、文字叠加图流入前端 |
| P-2 | **Aliyun OSS URL 格式错误**：`?x-oss-process=image/resize` 缺少尺寸参数（需 `resize,w_800`）→ CDN 返回 400 | 含该参数的图片全部加载失败 |
| P-3 | **商家相关性评分仅靠 URL 路径**：无法感知商家实际主营业务（如服装 vs 美妆 vs 户外） | 相关性评分不准，错误图片被优先显示 |
| P-4 | **`/assets/static/` 过滤过于激进**：部分商家将产品图存于此路径，导致好图被剔除 | 产品图被错误过滤，图片数量不足 |
| P-5 | **HEAD 检查对 CDN 逻辑矛盾**：`sensershop.com` 跳过 HEAD 检查，但 OSS URL 参数错误仍导致浏览器 400 | 跳过了检查但图片依然不可加载 |
| P-6 | **返回 40 张但前端只用 20**：浪费带宽，且没有质量保证 | 前端拿到的 40 张中质量参差不齐 |

---

### 9.3 重构设计方案（F-CRAWLER-001）

**新流程：**

```
后端爬取 → collectImages（目标 60 张原始候选，可继续爬更多）
         ↓
         [F-A] URL 预处理：修复 Aliyun OSS 参数 + 标准化 URL
         ↓
         [F-B] 硬过滤：黑名单（logo/icon/SVG/社交媒体/追踪像素）
         ↓
         [F-C] 业务相关性评分（基于商家主营业务关键词 + URL路径）
         ↓
         取评分 TOP 30，进行下载验证
         ↓
         [F-D] OCR 检测（tesseract.js，并行扫描）
               - 文字词数 > 15 → 丢弃（含文字图）
               - 文字词数 ≤ 15 → 保留
         ↓
         [F-E] 精选 TOP 20，补丁机制保底
         ↓
         返回精确 20 张 → 前端展示
```

---

### 9.4 各子方案详细设计

#### F-CRAWLER-001-A：Aliyun OSS URL 修复

**问题**：`https://xxx.sensershop.com/prd/image/xxx.jpg?x-oss-process=image/resize` 缺少 `w_800` → OSS 返回 400。

**方案**：在 URL 进入后续流程前，先做 URL 规范化：
- 检测 `x-oss-process=image/resize` 无尺寸参数 → 补全为 `x-oss-process=image/resize,w_800`
- 或直接移除 `x-oss-process` 参数（让 OSS 返回原图，通常更稳定）
- **推荐**：移除参数，让 CDN 返回原始图（无参数时 OSS 默认返回原图）

**实现位置**：`selectBestImages` 函数入口，新增 `normalizeImageUrl(url)` 工具函数。

---

#### F-CRAWLER-001-B：强化硬过滤规则

**新增过滤规则**：
- `/assets/static/` 路径：**改为降分**（score -= 30），不再直接丢弃（避免误杀产品图）
- 保留现有黑名单

---

#### F-CRAWLER-001-C：商家业务相关性评分增强

**方案**：从 `cache.pageText` + `cache.features` + `cache.navItems` 中提取商家主营业务关键词集合，对图片 URL 路径做关键词匹配加分。

**举例**：
- 商家是服装品牌 → pageText 含 "clothing, denim, jeans, apparel" → 图片路径含 `/clothing/` 或 `/denim/` 则加分 +30
- 商家是护肤品牌 → pageText 含 "skincare, moisturizer, serum" → 图片路径含 `/skincare/` 则加分 +30

**实现**：在调用 `selectBestImages` 时传入 `merchantContext`（包含 pageText 中前500字），函数内自动提取业务关键词并对图片评分加权。

---

#### F-CRAWLER-001-D：OCR 文字检测（核心新功能）

**方案**：使用项目已有依赖 `tesseract.js`，对 TOP 30 候选图片并行做 OCR，过滤含大量文字的图片。

**过滤规则**：
| OCR 检测结果 | 处理方式 |
|-------------|---------|
| 检测到词数 ≤ 5 个单词 | ✅ 保留（纯产品图） |
| 检测到词数 6-15 个单词 | ✅ 保留，但降分（可能有少量标签） |
| 检测到词数 > 15 个单词 | ❌ 丢弃（促销 banner / 文字海报） |
| OCR 超时（> 8s）| ✅ 保留（不影响正常流程） |

**性能策略**：
- 每批 5 张并行执行 OCR，最多处理 30 张
- OCR 并行与 AI 文案生成并行执行（不阻塞主流程）
- 超时兜底：若 OCR 总耗时 > 30s，直接返回 URL 过滤后的结果（不等 OCR）

**实现位置**：`selectBestImages` 函数中新增 `ocrFilterImages(urls[])` 步骤。

---

#### F-CRAWLER-001-E：精确返回 20 张 + 补丁机制

**方案**：
- OCR 过滤后，取评分 TOP 20 返回前端（`result.slice(0, 20)`）
- 若 OCR 后剩余 < 20 张：从 URL 过滤后的候选中补足（不做 OCR，作为兜底）
- 若总候选 < 20 张：当前已有的 `searchMerchantImages` 搜索兜底继续生效

**前端变化**：`MIN_IMAGES_TO_FRONTEND = 20`（已存在），返回值由 40 改为 20，前端图片槽位固定 20 张。

## 十一、F-CRAWLER-LOCALIZE-TLD — 多顶级域本地化探测（C-010，P0）

> 对应问题：商家填 `aerosus.nl`，国家填 BE，系统应自动识别并**静默切换**到 `aerosus.be`，不能继续把 `.nl` 当 BE 落地页。

### 11.1 现状与根因

现有 `crm-mvp/src/lib/crawl-pipeline.ts` 的 `normalizeLocaleInUrl(url, targetCountry)`（行 906）**只处理路径级 locale**：

- Case 1：`/en-sg/xxx` → `/en-be/xxx`（完整 `xx-yy` locale 前缀）
- Case 2：`/sg/xxx` → `/be/xxx`（纯 2 字母国家码前缀）
- **完全不处理 ccTLD**：`.nl` → `.be`、`.de` → `.at` 等顶级域本地化

aerosus 的架构恰好是独立 ccTLD（`aerosus.nl` / `aerosus.be` / `aerosus.de` / `aerosus.fr` 各自独立站），所以现有逻辑 100% 失效。

### 11.2 三层探测机制

| 层级 | 信号源 | 方法 | 命中率估计 | 可靠度 |
|------|--------|------|-----------|--------|
| L1 | `<link rel="alternate" hreflang="xx-YY">` | 抓主页 HTML，解析 `hreflang` 标签匹配 `xx-BE` / `nl-BE` / `fr-BE` | 80%（主流电商都遵循 Google 规范） | ★★★★★ |
| L2 | 页面内国家切换器 `<a href>` + 锚文本 | 搜 `België / Belgium / Belgique / BE` 等国家名关键字对应的 `<a>` | 15% | ★★★★ |
| L3 | ccTLD 硬切换 + HEAD 验证 | `aerosus.nl` → `aerosus.be`，发 HEAD 请求，200 且 `Content-Type: text/html` 才接受 | 5% 兜底 | ★★★ |

三层**按顺序执行，命中即停**；全部失败则退回原 URL（维持现状，不引入错误）。

### 11.3 实施点

**新增模块** `crm-mvp/src/lib/crawler-localize.ts`（独立文件，便于单测）：

```ts
export async function resolveLocalizedUrl(
  merchantUrl: string,
  targetCountry: string,
  proxyUrl?: string
): Promise<{ url: string; via: "hreflang" | "switcher" | "cctld" | "none" }>
```

**调用点**：`crawl-pipeline.ts` `buildCrawlCache()` 入口处，在 `crawlPage(merchantUrl, country)` **之前**调用。若命中则用新 URL 覆盖后续所有流程（爬取、sitelinks、图片、扩展），同时把"L1/L2/L3 命中原因"写进日志。

**静默替换策略（07 已确认）**：
- 前端不弹窗
- `buildCrawlCache` 返回结果新增 `resolvedUrl` 字段
- `generate-extensions/route.ts` 若 `resolvedUrl !== merchantUrl`，**直接 UPDATE `merchants.url`**（日志记录 before/after，可回溯）

### 11.4 边界与风险

| 风险 | 缓解 |
|------|------|
| hreflang 里写错（例如把 PL 的 URL 标成 hreflang="be"） | L1 命中后仍做 HEAD 验证，状态码非 200 拒绝 |
| 某些站把非本地化 URL 写在 switcher 里（广告位、子品牌） | L2 只接受**同主品牌域**（`aerosus.*` 主干名一致） |
| ccTLD 不存在导致 DNS 失败 | L3 用 5s 超时 HEAD，失败即放弃，不阻塞流程 |
| 静默替换后用户认为"我填的 URL 被偷改了" | 方案约定：**只改 merchants.url 的 TLD 部分**，不动其它路径；前端 Tooltip 显示"已自动切换到当地本地站"（弱提示，不是弹窗） |

### 11.5 回归验证清单（07 可补充）

- [ ] `aerosus.nl` + BE → 应切到 `aerosus.be`（L1 hreflang 命中）
- [ ] `aerosus.nl` + DE → 应切到 `aerosus.de`（L1）
- [ ] `aerosus.nl` + FR → 应切到 `aerosus.fr`（L1）
- [ ] `aerosus.nl` + NL → 保持 `aerosus.nl`（同国不切）
- [ ] 某个**不支持 BE** 的商家 + BE → 保持原 URL 不替换（三层均失败）
- [ ] 07 指定的 3~5 个其它多 ccTLD 商家（待 07 提供）

## 十二、F-CRAWLER-AEROSUS-NAV — 站内链接改走导航栏点击（C-011，P1）

> 对应问题：Sitelinks 生成失败。07 构思：按人工视角，识别导航栏 → 挑优质项 → "点击"拿到落地页。

### 12.1 现有流水线的"缺环"

`crm-mvp/src/lib/crawler.ts` 的 `crawlWithPuppeteerFull()`（行 869）**已经在抓** `navLinks`：

```
navLinks = document.querySelectorAll("nav a, header a, [role=navigation] a")
```

但目前这些 `navLinks` 只用于"图片补齐"和日志，**没流入 Sitelinks 生成**。`discoverSitelinkCandidates` 还在用 `getCommonProbePaths` 盲探服装词表 → 对 aerosus 这种汽车配件商全部 404。

### 12.2 改造点

1. **Puppeteer 强制触发条件放宽**（`crawler.ts` `crawlPage` 行 1530）
   - 现状：`if (links.length > 0 || images.length > 0)` → 认为 HTTP 成功，跳过 Puppeteer
   - 改为：`if (links.length >= 5 || images.length >= 10)` → 链接/图足够才算"真的成功"
   - 不足就继续走 Puppeteer 拿 `navLinks`

2. **`navLinks` 注入 sitelinks 候选**（`crawl-pipeline.ts` `discoverSitelinkCandidates`）
   - `buildCrawlCache` 新增把 `crawlWithPuppeteerFull` 的 `navLinks` 透传给 `discoverSitelinkCandidates`
   - 优先级：`puppeteerNavLinks(新) > pageLinks(原) > probePaths(废弃)`

3. **导航项优质度过滤**
   - 锚文本长度 2~40，且不在黑名单（Login / Sign in / Cart / My Account 等）
   - 同主域（去 www 后相同）
   - URL 去重后取前 6 条
   - 每条走 `fetchUrlMeta` 验证 200 + 非软 404

4. **保留一条兜底（可选，与 07 意见一致）**
   - `navLinks` 全军覆没时，可选保留**最小化 probe**（只试 `/contact`、`/about`、`/shipping`、`/returns` 这种所有电商都通用的路径，不用服装词表）

#### §2 L0 层 `fetchUrlMeta` 完美化

- 兼容旧调用点（外部传 `proxyUrl`）：若外部已传 `proxyUrl` 则优先用该 URL，否则按 country 动态轮换

##### §2.2 HTTP 全挂 → 批量 Puppeteer 兜底（共享 browser）

**文件**：`crm-mvp/src/lib/crawl-pipeline.ts` `discoverSitelinkCandidates`

- 第一轮：20 条 links 并发走强化后的 `fetchUrlMeta(url, undefined, country)`
- 若 candidates 仍 < 6，第二轮：
  - 收集所有 `ok=false` 的 URL（上限 8 条）
  - 新函数 `batchFetchMetaViaPuppeteer(urls, proxyUrl)`：
    - 用 `puppeteer-extra` 共享 1 个 browser（**不是每 URL 起一个**）
    - 并发 2 × 每条 `page.goto({ waitUntil: "domcontentloaded", timeout: 8000 })`
    - 抽取 `document.title` + `meta[name=description]` + `location.href`（同时得到 finalUrl）
    - 逐条 close page，全部完成后 close browser
    - 总预算：最坏 8 条 / 2 并发 × 8s = **32s** 额外开销（仅首次爬取，缓存命中则免）
  - 结果与第一轮同样经过 `isSoft404` / `isBlockedTitle` / `BAD_LINK_TEXTS` / `sanitizeAdText` 流程，**不降低质量门槛**

### 16.3 C-015 修复设计（本轮 PR）

07 原则依然是 "**只要 L0，做到 0 缺点**"。所以不新增 L1/L2 降级。L0 的语义扩展为"任何能拿到真实渲染页面的真实 HTTP 访问路径"，包括**无代理直连**（本服务器 SG 出口）——这**不是**降级到"信赖已爬链接"。

**§1：`fetchUrlMeta` 加无代理直连 fallback（最关键）**

- 文件：`crm-mvp/src/lib/crawler.ts`
- 流程：
  1. L0a：`country` 代理 + 6 UA IP 轮换（现状）→ 全部 `wasBlocked=true` 则进 L0b
  2. **L0b（新增）**：无代理，**3 UA** 再尝试一轮（SG 服务器出口直连）
  3. 若仍 `wasBlocked` → `ok=false`
- 为什么可以 fallback 到无代理：Google Ads 审核 bot 也是从 Google 全球节点（非目标国）来的，页面对 SG 出口可达 ≈ Google 审核可达，语义上仍是 L0 真实访问。
- 副作用：+3 次 HTTP 请求（每次 12s timeout），但只在 L0a 全灭时触发，绝大多数商家不命中此分支。

**§2：`batchFetchMetaViaPuppeteer` 增强**

- 文件：`crm-mvp/src/lib/crawler.ts`
- 增加：
  1. **Cloudflare 挑战页标题黑名单**：`Just a moment`、`Attention Required`、`Checking your browser`、`One moment, please` → 视为 `ok=false, isSoft404=true`
  2. **JS 挑战等待**：`domcontentloaded` 后调 `page.waitForFunction('document.title && !/just a moment|attention required|checking/i.test(document.title)', { timeout: 6000 })`，给 CF challenge 窗口
  3. **proxy 失败 fallback**：若有 `proxyUrl` 且整个 batch 产出 0 条 ok，自动**再跑一次无代理 batch**
  4. **失败原因 log**：每条 URL 记录 `status/title/elapsed`

**§3：TLD 切换时同步剥离 `merchant_name` 末尾国家后缀**

- 文件：`crm-mvp/src/app/api/user/ad-creation/generate-extensions/route.ts`
- 位置：TLD 本地化静默替换分支（C-014 §1 写的那段）
- 规则：`merchant_name` 末尾若匹配 `\s+(NL|BE|DE|FR|UK|GB|IT|ES|NL|AT|CH|DACH|EU|USA|US|CA|AU|JP|KR|CN)` → 去掉该后缀；同步 UPDATE `user_merchants.merchant_name`
- 示例：
  - "Aerosus NL" + aerosus.nl → aerosus.be (BE) → **"Aerosus"**
  - "Aerosus DACH" + aerosus.de → aerosus.at (AT) → **"Aerosus"**
  - "Brand Name" + 无 TLD 切换 → 不改
- 传给 AI 的 `merchantName` 变量一并更新为剥后版本（即使 DB 更新失败也生效）。

**§4：关键诊断日志 `console.log` → `console.warn`**

- 文件：`crm-mvp/src/lib/crawl-pipeline.ts`、`crm-mvp/src/lib/crawler.ts`
- 目标：让 Next.js 16 prod 生产下 PM2 能捕获到：
  - `[Sitelinks] 使用 ... 代理探查候选链接`
  - `[Sitelinks] HTTP 候选 N 条，HTTP ok=X 失败=Y，Puppeteer 兜底尝试=Z，最终 M 候选`
  - `[Sitelinks] Puppeteer 兜底异常`
  - `[Crawler] fetchUrlMeta 代理全灭，回退无代理直连`
- 只提升**关键汇总行**级别，不改 debug 行，避免日志噪音。

## 十七、C-016 广告生成流程重建（2026-04-18）

### 17.1 背景

07 多轮反馈（2026-04-17 / 04-18）暴露的质量问题与架构性问题：

1. **V-1 TLD 本地化反复回退**：`aerosus.nl` 投 BE 国家时，三级探测（代理 / 直连 / Puppeteer）全部被 CF / WAF 阻断，`resolvedMerchantUrl` 字段取不到值。
2. **V-2 Sitelinks 全空**：`aerosus.be` 爬不到导航链接，主流程未做扩源兜底。
3. **V-3 否定关键词匮乏**：已在 C-014 通过扩充词库解决，C-016 不再涉及。
4. **V-4 标题泄漏 "NL"**：`user_merchants.merchant_name = "Aerosus NL"` 被原样喂给 AI prompt，BE 投放出现 "Aerosus NL - 2 Jaar Garantie"。
5. **架构性污染**：`generate-extensions/route.ts` 在切换 TLD 时直接 `prisma.user_merchants.update(merchant_url, merchant_name)` —— 违反"平台 API 数据不可污染"的原则，且用户投放多国时会相互覆盖。
6. **合规闭环缺口**：现有 `complianceAutoFix` 只基于 `aiRuleProfile`（用户规则），没有 Google 官方政策的多语言硬规则，也没有"事实索赔对爬取证据"的验证。
7. **高流畅体感不足**：合规返工时若内部同步重试完才 send，前端会有 5-8 秒空窗。

| **C. 独立 URL-only 管线** | 新建 `/api/user/articles/crawl-by-url` + `POST /api/user/articles/by-url`，复用 crawler 但不写 `user_merchant_id` | 隔离清晰 | 重复代码多；未来维护两套 |

**推荐方案 A**，原因：
- schema 只加 1 列（`user_merchants.source`），迁移零风险
- 爬虫/AI/发布链路 100% 复用，没有新管线维护成本
- 可在「我的商家」列表用 `source='url_direct'` 过滤掉，不打扰日常 UI

## 二十一、C-021 站点选择器支持输入搜索

> 需求原文："**现在的站点选择只支持翻找，不支持输入域名查找，很麻烦，需要优化**"

### 21.1 现状

`publish/page.tsx` 含 **2 处** `sites` / `activeSites` 的 Select：

| 位置 | 行号（prod）| 现状 props | 问题 |
|------|-----------|-----------|------|

| Step 5 的"发布到站点" | 851-852 | 同上 | 同上 |

**此外需排查**：

- `/admin/sites` 管理后台站点列表（已有独立页面，不在本次范围）
- `/user/articles` 文章管理页（若有按站点筛选的 Select 同步优化）

### 21.2 改动方案（最小侵入）

**通用做法**：给 Select 加三个 props

```tsx
<Select
  showSearch
  placeholder="搜索站点名或域名"
  filterOption={(input, option) => {
    const s = input.toLowerCase().trim();
    if (!s) return true;
    const site = activeSites.find((x) => x.id === option?.value);
    if (!site) return false;
    return (
      site.site_name.toLowerCase().includes(s) ||
      site.domain.toLowerCase().includes(s)
    );
  }}
  options={activeSites.map((s) => ({ value: s.id, label: `${s.site_name} (${s.domain})` }))}
  // ... 原有 onChange / value 等不动
/>
```

**边界**：
- 38 站点全量加载，纯前端过滤，无需新 API
- `filterOption` 同时按 `site_name` + `domain` 做子串匹配，不区分大小写
- 输入支持去首尾空格（避免复制粘贴带空格）
