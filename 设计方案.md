## 设计原则

### F-01 数据统计口径
- **总览指标（总花费/总佣金/ROI）** = 全部去重 Campaign 聚合，不受表格行数限制
- **表格行** = 最多 200 行，按状态排序（启用优先），合计行数据与总览一致
- 前端增加口径说明，当行数被截断时显示提示
- 佣金明细提供"按商家"和"按平台账号"两个维度，均为独立聚合

### F-02 广告图片 → 文章图片同步
- `ad_creatives.image_urls` 是广告最终选图的权威来源
- 广告提交成功时同步到 `articles.images`
- 文章 AI 生成阶段仅在文章无图时填入爬虫图（初始占位）
- 共享布局工具 (`article-image-layout.ts`) 统一管理图片在 HTML 中的位置

### F-03 发布一致性
- **预览即最终**：文章发布页预览的 HTML 就是最终发布到站点的内容
- 发布器只允许：路径规范化 + 图片本地化（下载外链/读取本地上传图到远程站点）
- 发布器不允许：自动补图、重排图片位置、改变图片内容或顺序
- 发布前先保存最终 content 到数据库，确保发布器读到的是用户确认的版本

## F-04 广告创建优化设计原则

### 图片上传修复
- 搜索广告系列（SEARCH Campaign）必须使用 `AD_IMAGE` field_type
- `MARKETING_IMAGE` 仅适用于展示广告/Performance Max，搜索广告会报 `FIELD_TYPE_INCOMPATIBLE_WITH_CAMPAIGN_TYPE`
- 图片仍通过 `campaign_asset_operation` 挂载到 Campaign 级别

### AI 文案质量分优化
- 采用07提供的三步提示词策略重构 AI prompt
- 标题规则：第一条品牌关联、折扣和物流各一条并优先展示、控制大写字符、≤30字符严格核查
- 描述规则：有且仅有一条同时包含折扣+物流信息、50-90字符范围、品牌特色语言
- 所有文案必须符合 Google Ads 平台规范，不得欺骗误导消费者

### 关键词匹配类型
- 支持三种匹配类型：广泛匹配(BROAD)、词组匹配(PHRASE)、完全匹配(EXACT)
- 每个关键词 Tag 旁有独立的下拉选择器，可随时切换
- 新增关键词时可预选匹配类型，默认词组匹配
- 后端 submit 已支持读取 matchType，无需额外修改

### 广告扩展（新增4种）
- **促销扩展 (Promotion)**：支持百分比/金额折扣、促销代码、促销场合（黑五/圣诞等15种）、货币选择
- **价格扩展 (Price)**：支持9种价格类型、最多8个价格项（标题+描述+价格+URL）
- **致电扩展 (Call)**：支持8个国家代码 + 电话号码
- **结构化摘要 (Structured Snippet)**：支持13种标题类型、3~10个值
- 所有扩展通过 `campaign_asset_operation` 挂载到 Campaign 级别

### 二轮验收修正（本地化 + 质量）
- 促销/价格/致电/结构化摘要的默认提交字段必须按目标国家自动兜底，禁止再次回落到固定 `en`、`USD`、`US`
- 结构化摘要 header 内部保持 Google Ads 兼容标准值，前端只做展示层本地化，避免因本地化字符串导致提交不兼容
- Callout 若同时命中配送与退货政策，优先合并为一条 ≤25 字文案，并在 AI 输出后再次做去重与归一化
- 标题与描述必须过滤显式过期日期、重复语义与无意义填充语，且保留商业吸引力与品牌相关性

### 图片 OCR 文字检测
- 使用 tesseract.js（纯 JS OCR 引擎）检测图片中的文字
- 图片上传后自动触发 OCR 检测，含文字图片标红警告
- 过滤低置信度噪声（confidence < 50），至少2个有意义词才判定为含文字
- 图片区域增加明确提示：需与品牌强关联、不要有文字

## 关键词政策过滤漏洞修复（CHG-065）

**问题**：07 反馈"Martian Mushrooms"商家的广告创建中仍出现违法关键词：`buy mushrooms magic`、`buy magic mush`、`grow kit mushroom magic`、`psilocybe mushroom growing kit`。

**根因分析**：
1. **正则词序固定**：原 `magic\s*mushroom` 仅匹配 "magic" 在前的情况，`buy mushrooms magic`（magic 在后）漏检
2. **缩写未覆盖**：`magic mush` 缩写形式和 `psilocybe`（裸盖菇属名）未加入正则
3. **存量数据无过滤**：SemRush 旧数据已存入数据库，从 DB 读取时（`mapStoredKeywordsForClient`）完全没有政策过滤
4. **提交无拦截**：即使前端显示了违规关键词，提交到 Google Ads 时也没有最后一道防线

**修复方案（全链路 4 层拦截）**：

| 层级 | 文件 | 修改内容 |
|------|------|----------|
| 1. SemRush 入库过滤 | `keyword-optimizer.ts` | 将 `POLICY_RISK_PATTERNS` 拆分为 `POLICY_RISK_SINGLE_PATTERNS`（单词级正则）+ `POLICY_RISK_COMBO_SETS`（词序无关组合检测），提取统一函数 `isPolicyRiskKeyword()` 并导出 |
| 2. 存量数据过滤 | `ad-keyword-pipeline.ts` | 在 `mapStoredKeywordsForClient` 中增加 `.filter(r => !isPolicyRiskKeyword(r.keyword_text))`，从 DB 读出时即过滤 |
| 3. 违规检查增强 | `ai-rule-profile.ts` | 修复 `BASIC_POLICY_RISK_PATTERNS` 管制物质正则（去掉固定词序的 `magic\s*mushroom`，新增 `psilocybe`），在 `checkItemViolations` 和 `collectAiRuleViolations` 中增加 `isPolicyRiskKeyword` 兜底检查 |
| 4. 提交硬拦截 | `submit/route.ts` | 在构造 `ad_group_criterion_operation` 前调用 `isPolicyRiskKeyword`，违规关键词直接 `continue` 跳过，不发送给 Google |

**核心设计 — `isPolicyRiskKeyword` 函数**：
- 统一导出自 `keyword-optimizer.ts`，所有路径复用同一函数
- 两层检测：单词正则 → 多词组合（词序无关）
- 组合检测使用 `phraseContainsAllRoots`：只要短语中同时包含所有词根即判定违规

**验证结果**：

| 关键词 | 结果 | 匹配方式 |
|--------|------|----------|
| `buy mushrooms magic` | BLOCKED | 组合检测（magic + mushroom） |
| `buy magic mush` | BLOCKED | 组合检测（magic + mush） |
| `grow kit mushroom magic` | BLOCKED | 组合检测（magic + mushroom） |
| `psilocybe mushroom growing kit` | BLOCKED | 单词正则（psilocybe） |
| `magic mushroom grow kit` | BLOCKED | 组合检测（magic + mushroom） |
| `mushroom growing kit` | ALLOWED | 合法关键词，放行 |
| `mycology supplies` | ALLOWED | 合法关键词，放行 |
| `mushroom spores UK` | ALLOWED | 合法关键词，放行 |

## 商家状态与广告系列状态不对齐修复（CHG-066）

**问题**：07 反馈商家管理页面的商家状态与数据中心的广告系列状态不一致。广告系列状态是正确的，但商家状态显示错误。

**根因分析**：
1. **重复广告记录导致状态不一致**：数据库中同一个 `google_campaign_id` 可能存在多条 `campaigns` 记录（广告创建时建一条 + MCC 同步时发现同一广告又建一条）。MCC 同步（`syncUserCampaignStatuses`）按 `google_campaign_id` 索引更新，只会更新其中一条的 `google_status`。但商家页面的 `campaignMap` 没有按 `google_campaign_id` 去重，可能选中了未更新的那条旧记录，导致显示过时状态。数据中心页面有去重逻辑，所以能正确显示。
2. **商家页面不自动刷新状态**：商家 API 的 GET 处理器中，`autoLinkAndClaimMerchants`（自动关联+状态同步）只在 `consistClaimedCount === 0`（用户无已领取商家）时才触发。wj07 有 65 个商家，该函数永远不会被调用，导致页面只显示数据库中的旧状态。

**修复方案**：

| 文件 | 修改内容 |
|------|----------|
| `merchants/route.ts` | 1. campaignMap 构建前先按 `google_campaign_id` 去重（保留 `last_google_sync_at` 最新的记录），与数据中心去重逻辑一致 |
| `merchants/route.ts` | 2. 移除 `consistClaimedCount === 0` 条件，每次加载商家页面时都调用 `autoLinkAndClaimMerchants`（该函数内部对无变化场景做了短路返回，不影响性能） |
| `campaign-merchant-link.ts` | 3. `syncMerchantStatusFromCampaigns` 函数内部也增加按 `google_campaign_id` 去重，避免重复记录影响 `user_merchants.status` 的判定 |
