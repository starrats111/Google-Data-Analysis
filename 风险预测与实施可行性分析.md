## 一、OPT-001 全局消息提醒系统

## 二、OPT-002 拒付佣金变动检测

### 2.1 风险预测（未来可能出现的问题）

| 风险编号 | 风险描述 | 触发条件 | 影响程度 | 缓解措施（方案中已含/建议） |
|----------|----------|----------|----------|----------------------------|
| R-002-1 | 与 backfill 同时写库导致 SQLite 锁 | 06:30 检测与 06:00 backfill 重叠（backfill 拖到 06:30 后） | 高 | 方案已设 06:30 + 前置检查「backfill 是否仍在跑」 |
| R-002-2 | user_id 为 NULL 的订单被漏算 | affiliate_transactions 中部分记录 user_id 为空 | 中 | 方案已要求通过 affiliate_account_id 反查 user_id，无法关联则跳过并打日志 |
| R-002-3 | 快照与通知不同步 | 写快照成功、写通知失败（如磁盘满） | 高 | 方案已要求同一事务内写通知+快照（M-05） |
| R-002-4 | 阈值 100 USD 对部分用户不合适 | 大客户 100 波动属正常，小客户 100 已很大 | 中 | 方案已配置化 REJECTED_COMMISSION_DAILY_THRESHOLD，后续可考虑按用户配置 |
| R-002-5 | 上月变动通知过于频繁 | 联盟平台频繁修正上月数据，导致每天都有「上月变动」 | 低 | 可接受；若需收敛可加「同一 period 仅提醒一次」或冷却期 |

### 2.2 实施可行性模拟

| 模拟项 | 结论 | 说明 |
|--------|------|------|
| 数据来源 affiliate_transactions | 可行 | 表存在，有 transaction_time、status、commission_amount、user_id；需确认 nullable 与反查逻辑 |
| 定时任务 06:30 与 backfill 错峰 | 可行 | scheduler 中新增 job，并加「backfill 是否运行中」判断 |
| 事务边界 | 可行 | 使用同一 Session、先写通知再写快照，同一 commit |
| 新表 commission_snapshots | 可行 | 新表 + create_all，与 OPT-001 一起在 startup 建表 |

**结论：** 可实施。关键依赖：事务内写通知+快照、06:30 错峰与前置检查、user_id 为 NULL 时的反查与日志。

## 三、OPT-003 已付佣金展示 / OPT-004 ROI 公式修改

### 3.1 风险预测（未来可能出现的问题）

| 风险编号 | 风险描述 | 触发条件 | 影响程度 | 缓解措施（方案中已含/建议） |
|----------|----------|----------|----------|----------------------------|
| R-003-1 | 历史报表与当前 ROI 口径不一致 | 报表中历史 ROI 按旧公式算，页面按新公式 | 低 | 方案已有 ROI Tooltip 说明公式；历史报表可保留旧口径或加说明 |
| R-004-1 | 费用为 0 时除零 | (佣金-费用)/费用 在费用=0 时无定义 | 中 | 实现时需判断：费用≤0 时显示「-」或「N/A」，不计算 ROI |
| R-004-2 | 用户对比历史数据困惑 | 同一时间段新旧公式结果不同 | 低 | Tooltip 写明「ROI = (佣金-费用)÷费用」 |

### 3.2 实施可行性模拟

| 模拟项 | 结论 | 说明 |
|--------|------|------|
| 已付佣金数据源 | 可行 | platform_data API 已返回 approved_commission，前端仅展示与汇总 |
| 明细模式 status===approved | 可行 | 明细为逐条交易，按 status 渲染已付列，逻辑明确 |
| ROI 公式与费用=0 | 需实现时处理 | 设计方案未显式写「费用为 0」分支，建议实现时统一：费用≤0 不展示数值，仅展示「-」或 Tooltip 说明 |

**结论：** 可实施。唯一需在设计上补一句：**ROI 在费用≤0 时的展示规则（建议：不计算，显示「-」或「N/A」）。**

## 四、OPT-005 MCC 数据同步双模式（API / 脚本）

### 4.1 风险预测（未来可能出现的问题）

| 风险编号 | 风险描述 | 触发条件 | 影响程度 | 缓解措施（方案中已含/建议） |
|----------|----------|----------|----------|----------------------------|
| R-005-1 | Google Ads 脚本未按时跑完 | 用户未配置定时或脚本报错，04:30 拉 Sheet 时无昨日数据 | 高 | 后端读取时校验「最新一行 date」是否为昨日；非昨日则记 last_sync_status=异常并告警/日志 |
| R-005-2 | Sheet 列名或顺序与模板不一致 | 用户改表头或删列 | 高 | 读取时按列名映射，缺列则该列填默认值或跳过；必列缺失则本次同步失败并记录原因 |
| R-005-3 | 同一 MCC 同时走 API 与脚本 | 误配置或定时重叠，导致重复写入 | 低 | 表唯一约束 mcc_id+campaign_id+date，upsert 可去重；建议脚本模式 MCC 不再调 API |
| R-005-4 | 服务账号无 Sheet 权限 | 用户未把表格共享给服务账号邮箱 | 高 | 已有「测试 Sheet 连接」API；失败时明确提示「请将表格共享给 xxx@xxx.iam.gserviceaccount.com」 |
| R-005-5 | Sheets API 配额超限 | 多 MCC、多次读取/写入 | 低 | 配额远高于需求；若未来 MCC 数很多可加限流 |
| R-005-6 | 脚本模板与后端解析不一致 | 脚本输出列名/格式与后端映射表不一致 | 高 | 脚本模板由后端生成，列名写死为方案中的英文列名；后端按同一套列名解析，避免用户手改 |
| R-005-7 | 回滚后 script 模式 MCC 被当 API 再调 | 见实施前确认清单第 5 项 | 高 | 需 07 在实施前确认清单中选定回滚策略（停用/兼容/接受失败） |

### 4.2 实施可行性模拟

| 模拟项 | 结论 | 说明 |
|--------|------|------|
| google_mcc_accounts 新增 4 字段 | 可行 | 需手动 ALTER 或迁移脚本；实施前确认清单第 1 项待 07 确认迁移方式 |
| 定时任务按 sync_mode 分支 | 可行 | sync_google_ads_data_job 内遍历 MCC，api 走现有 sync，script 走 GoogleSheetSyncService；若选「04:30 单独调度」需新增一条定时 |
| CampaignMatcher 复用 | 可行 | 已有 extract_platform_from_campaign_name(campaign_name, user_id)，Sheet 每行有 campaign_name，写入前调用即可得到 extracted_platform_code/account_code |
| gspread + google-auth | 可行 | 需 pip 安装；认证复用现有服务账号，需在 Google Cloud 开启 Sheets API 并授权只读范围 |
| 脚本模板列名与后端一致 | 可行 | 后端生成脚本时写死列名（date, customer_id, campaign_id, ...），后端解析按同列名读取，避免不一致 |

**结论：** 可实施，但依赖 07 对「实施前确认清单」5 项的确认（迁移方式、04:00/04:30 调度、GOOGLE_SHEET_SYNC_ENABLED、服务账号、回滚策略）。实施时需明确：**费用=0 时 ROI 展示规则**、**Sheet 缺列/列名不符时的失败与提示**。

## 五、跨功能风险与实施顺序建议

| 风险 | 涉及功能 | 建议 |
|------|----------|------|
| 数据库 startup 建表顺序 | OPT-001、OPT-002 | 在 main.py startup 中先 import Notification、CommissionSnapshot，再 create_all，保证两张新表同时存在 |
| 定时任务时间线集中 | OPT-002(06:30)、backfill(06:00)、OPT-005(04:00/04:30) | 保持 06:30 拒付检测、04:00 主同步；若 OPT-005 选「04:30 单独读 Sheet」，需避免与 04:00 主任务抢库锁，可接受 |
| 前端 Layout 变更与多页面 | OPT-001 | 铃铛在 Layout 中一次接入，所有页面即生效，无重复劳动 |

**实施顺序建议：**  
1）OPT-003、OPT-004（纯前端、风险小，可先上）；  
2）OPT-001 + OPT-002（通知+检测，一起上）；  
3）OPT-005（依赖实施前确认清单 5 项确认后再动代码）。

## 六、设计上需 07 确认的事项（Plan 模式）

以下为设计层面的确认，确认前**不落代码**。请 07 逐项确认或给出偏好。

### 6.1 OPT-004：ROI 在费用≤0 时的展示

- **现状：** 方案未明确当「费用=0 或费用为空」时，ROI 卡片与表格如何展示。
- **建议：** 费用≤0 或非数字时，ROI 不计算，卡片与表格中显示「-」或「N/A」，Tooltip 可写「费用为 0 时无法计算 ROI」。
- **请确认：** 是否采用上述规则？或你希望的表现形式（例如显示 0、显示「∞」等）？

### 6.2 OPT-005：Sheet 数据不完整时的策略

- **场景：** 后端读取 Sheet 时，发现「最新一行日期不是昨天」或「必列缺失/列名不符」。
- **建议：**  
  - 若最新行不是昨天：本次同步视为失败，更新该 MCC 的 last_sync_status=异常，last_sync_message=「Sheet 中无昨日数据，请检查 Google Ads 脚本是否已运行」。  
  - 若必列缺失或列名不符：本次同步失败，last_sync_message 中写明缺失列名或「列名与模板不一致，请使用系统生成的脚本」。
- **请确认：** 是否按上述策略做？是否需要额外通知方式（如写入通知中心、仅日志）？

### 6.3 OPT-005：实施前确认清单 5 项

- 实施前确认清单中的 **5 项**（数据库迁移方式、04:00/04:30 调度、GOOGLE_SHEET_SYNC_ENABLED、服务账号复用、回滚时 script 模式 MCC 行为）是否已全部确认？  
- 若尚未填写，建议在动 OPT-005 代码前先由 07 在 [实施前确认清单.md](实施前确认清单.md) 中勾选/填写，再按确认结果实施。

### 6.4 可选：OPT-001 经理端「按员工筛选」通知

- **现状：** 经理看「全部通知」，仅靠分页；员工多时列表较长。
- **建议：** 本期可不做；若后续需要，可在下拉面板增加「按员工筛选」下拉框（仅经理可见）。
- **请确认：** 本期是否不做「按员工筛选」？若无异议，按「本期不做」执行。
