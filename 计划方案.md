## 4. 核心设计（按问题编号）

### A. 预算分析 30 天逐日数据查询规格（M-28）

#### A1. 问题分析

设计方案 Section 2.6.1 预算效率分析需要"过去 30 天各 campaign 的日花费 vs 日佣金数据"用于拟合花费-佣金关系曲线并计算边际效率。但当前设计仅在 Section 2.3.4 定义了 14 天逐日花费查询（用于异常检测），缺少：

1. **30 天日花费查询**（14 天不够，需扩展到 30 天）
2. **30 天日佣金查询**（完全缺失，无法计算 `delta_commission / delta_cost`）
3. **佣金→campaign 映射**（佣金在 `AffiliateTransaction`，广告在 `GoogleAdsApiData`，两者通过 MID 关联）

#### A2. 补充查询规格

**查询 1：30 天日花费（从 `GoogleAdsApiData`）**

```python
daily_cost_30d = db.query(
    GoogleAdsApiData.campaign_id,
    GoogleAdsApiData.campaign_name,
    GoogleAdsApiData.date,
    GoogleAdsApiData.cost,
    GoogleAdsApiData.clicks,
    GoogleAdsApiData.impressions,
    GoogleAdsApiData.cpc,
).filter(
    GoogleAdsApiData.customer_id.in_(customer_ids),
    GoogleAdsApiData.date >= end_date - timedelta(days=29),
    GoogleAdsApiData.date <= end_date,
).order_by(GoogleAdsApiData.campaign_name, GoogleAdsApiData.date).all()
```

输出格式：

```python
{
    "campaign_name": [
        {"date": "2026-02-05", "cost": 45.2, "clicks": 102, "cpc": 0.44},
        ...  # 30天逐日
    ]
}
```

**查询 2：30 天日佣金（从 `AffiliateTransaction`）**

```python
daily_commission_30d = db.query(
    AffiliateTransaction.merchant_id,
    func.date(AffiliateTransaction.transaction_time).label("date"),
    func.sum(AffiliateTransaction.commission_amount).label("daily_commission"),
    func.count(AffiliateTransaction.transaction_id).label("daily_orders"),
```

输出格式：

```python
{
    "mid_12345": [
        {"date": "2026-02-05", "commission": 32.10, "orders": 8},
        ...  # 30天逐日
    ]
}
```

**查询 3：MID → Campaign 映射（复用现有 `CampaignPlatformMapping`）**

```python
mid_campaign_map = {}
for mapping in db.query(CampaignPlatformMapping).filter(
    CampaignPlatformMapping.campaign_name.in_(campaign_names),
).all():
    mid_campaign_map[mapping.campaign_name] = mapping.merchant_id
```

通过 `mid_campaign_map` 将日佣金数据关联到对应 campaign，实现花费-佣金曲线拟合。

#### A3. 数据不足处理

| 情况 | 处理 |
|------|------|
| campaign 无对应 MID 映射 | 该 campaign 跳过预算分析，标记 `"skipped": true, "reason": "无MID映射"` |
| MID 有映射但佣金数据不足 15 天 | 仍参与分析，但标记 `"data_quality": "limited"` |
| 日花费全为 0 | 跳过，标记 `"skipped": true, "reason": "30天内无花费"` |

### B. CPA 诊断上期 orders/commission 查询（M-29）

#### B1. 问题分析

Section 2.5.3 的 CPA 归因算法需要 `old_cpa = previous_cost / previous_orders`（上期 7 天的 CPA）。当前 `merchant_l7d_data` 仅查询当期 7 天（`begin_date ~ end_date`），上期数据（`end_date-13 ~ end_date-7`）没有查询入口。

#### B2. 补充查询规格

**方案：扩展为双窗口查询，复用 `merchant_l7d_data` 逻辑。**

在 `api_analysis_service.py` 的 `generate_l7d_analysis()` 方法中，现有 `merchant_l7d_data` 查询（L772-804）之后新增上期查询：

```python
# 上期窗口日期
prev_begin = end_date - timedelta(days=13)
prev_end = end_date - timedelta(days=7)

merchant_previous_data = {}
prev_txns = db.query(
    AffiliateTransaction.user_id,
    AffiliateTransaction.merchant_id,
    func.sum(AffiliateTransaction.commission_amount).label("total_commission"),
    func.count(AffiliateTransaction.transaction_id).label("total_orders"),
    func.count(func.distinct(func.date(AffiliateTransaction.transaction_time))).label("order_days"),
).filter(
    AffiliateTransaction.transaction_time >= prev_begin,
    AffiliateTransaction.transaction_time <= prev_end,
    AffiliateTransaction.merchant_id.isnot(None),
    AffiliateTransaction.merchant_id != "",
).group_by(
    AffiliateTransaction.user_id,
    AffiliateTransaction.merchant_id,
).all()

for txn in prev_txns:
    merchant_previous_data[(txn.user_id, txn.merchant_id)] = {
        "commission": float(txn.total_commission or 0),
        "orders": int(txn.total_orders or 0),
        "order_days": int(txn.order_days or 0),
    }
```

#### B3. CPA 归因中的使用方式

```python
mid = campaign_data.get(campaign_key, {}).get("mid")

# 当期
current_l7d = merchant_l7d_data.get((user_id, mid), {})
current_orders = current_l7d.get("orders", 0)
current_commission = current_l7d.get("commission", 0)

# 上期
previous_l7d = merchant_previous_data.get((user_id, mid), {})
previous_orders = previous_l7d.get("orders", 0)
previous_commission = previous_l7d.get("commission", 0)

# CPA 计算（防除零）
current_cpa = current_cost / current_orders if current_orders > 0 else None
previous_cpa = previous_cost / previous_orders if previous_orders > 0 else None
```

#### B4. 数据不足处理

| 情况 | 处理 |
|------|------|
| 上期 orders = 0 | `old_cpa = None`，该 campaign 跳过 CPA 归因，标记 `"excluded_reason": "上期无订单数据"` |
| 当期和上期 orders 均 = 0 | 跳过 CPA 诊断整体，标记 `"cpa_diagnosis": null, "skip_reason": "当期与上期均无订单"` |
